# Plan: Migrate store.ts from better-sqlite3 to node:sqlite (DatabaseSync)

Replace the native `better-sqlite3` dependency in `src/store.ts` with Node's built-in `node:sqlite` (`DatabaseSync`/`StatementSync`). The public `Store` API and SQLite schema are unchanged — this is a pure engine swap.

## Step 3 — src/store.ts: tx() method (critical)

Current (lines 325-327):
```ts
tx<T>(fn: () => T): T {
  return this.db.transaction(fn).immediate();
}
```

`node:sqlite` has no `.transaction()` helper. Replace with a manual BEGIN IMMEDIATE wrapper:
```ts
/**
 * Run `fn` in a `BEGIN IMMEDIATE` transaction (write lock acquired up front).
 * Returns fn's result; rolls back and rethrows if fn throws.
 * Never call tx() re-entrantly — node:sqlite does not support nested transactions.
 */
tx<T>(fn: () => T): T {
  this.db.exec('BEGIN IMMEDIATE');
  try {
    const result = fn();
    this.db.exec('COMMIT');
    return result;
  } catch (err) {
    this.db.exec('ROLLBACK');
    throw err;
  }
}
```

### Why BEGIN IMMEDIATE and not plain BEGIN

liveloop uses a commit-fingerprint compare-and-swap (CAS) to serialize concurrent engine ticks across processes. `BEGIN IMMEDIATE` acquires the SQLite write lock at transaction start, so two concurrent processes cannot both enter the CAS read phase of the same `tx()` simultaneously. With plain `BEGIN` (DEFERRED), SQLite defers the write lock until the first write statement — two processes can both read the fingerprint before either writes, and then race at the write, producing a `SQLITE_BUSY` mid-transaction rather than at the cleaner `BEGIN` boundary. liveloop's engine.ts callers do not retry inside `tx()` — they rely on the busy_timeout to serialize at the `BEGIN` gate. Switching to plain BEGIN would silently break the CAS guarantee under concurrent writers.

### Nesting note

Under `better-sqlite3`, `.transaction()` handled re-entrant calls via SQLite savepoints. The manual wrapper above throws `cannot start a transaction within a transaction` if `tx()` is called re-entrantly. A grep confirms `tx()` is called from 13 sites in `engine.ts`, all at the top level — never inside a `tx()` callback. No code change needed, but the JSDoc must document this constraint.

### Test A: CAS fingerprint-check (single process, two Store instances)

```ts
test('tx() CAS: second writer detects fingerprint change and does not commit', () => {
  const dir = mkdtempSync(join(tmpdir(), 'liveloop-cas-'));
  const dbPath = join(dir, 'cas.db');
  try {
    const s1 = new Store(dbPath);
    const wf = 'wf_cas';
    const base: ArtifactData = {
      workflow: wf, path: 'plan', producer: 'planner',
      acceptance: 'green', version: 1,
      fingerprint: { plan: 1 },
      reasons: [], judgmentRejects: 0, schemaRejects: 0,
    };
    s1.putArtifact(base);
    const s2 = new Store(dbPath);

    // Both read fingerprint before any tx() — both see { plan: 1 }
    const fp1 = s1.getArtifact(wf, 'plan')!.fingerprint;
    const fp2 = s2.getArtifact(wf, 'plan')!.fingerprint;
    assert.deepEqual(fp1, { plan: 1 });
    assert.deepEqual(fp2, { plan: 1 });

    // s1 wins: commits with new fingerprint
    let s1Won = false;
    s1.tx(() => {
      const cur = s1.getArtifact(wf, 'plan')!.fingerprint;
      assert.deepEqual(cur, { plan: 1 });
      s1.putArtifact({ ...base, fingerprint: { plan: 2 }, version: 2 });
      s1Won = true;
    });

    // s2 loses: fingerprint no longer matches its stale read
    let s2Won = false;
    let s2Err: unknown;
    try {
      s2.tx(() => {
        const cur = s2.getArtifact(wf, 'plan')!.fingerprint;
        if (JSON.stringify(cur) !== JSON.stringify(fp2)) {
          throw new Error('CAS conflict');
        }
        s2.putArtifact({ ...base, fingerprint: { plan: 3 }, version: 3 });
        s2Won = true;
      });
    } catch (e) { s2Err = e; }

    assert.ok(s1Won, 's1 must have committed');
    assert.ok(!s2Won, 's2 must not have committed');
    assert.ok(s2Err instanceof Error, 's2 must have thrown');
    // Only s1 commit visible
    assert.deepEqual(s1.getArtifact(wf, 'plan')!.fingerprint, { plan: 2 });

    s1.close();
    s2.close();
  } finally {
    rmSync(dir, { recursive: true, force: true });
  }
});
```

### Test B: BEGIN IMMEDIATE serialization (raw DatabaseSync, two connections)

This test is what fails if `tx()` uses plain `BEGIN` instead of `BEGIN IMMEDIATE`.

```ts
test('tx() BEGIN IMMEDIATE: second connection is blocked at BEGIN, not mid-write', () => {
  const dir = mkdtempSync(join(tmpdir(), 'liveloop-imm-'));
  const dbPath = join(dir, 'imm.db');
  try {
    const db1 = new DatabaseSync(dbPath);
    const db2 = new DatabaseSync(dbPath);
    db1.exec('PRAGMA journal_mode = WAL');
    db2.exec('PRAGMA journal_mode = WAL');
    db1.exec('PRAGMA busy_timeout = 100');
    db2.exec('PRAGMA busy_timeout = 100');
    db1.exec('CREATE TABLE t (x INTEGER)');

    // db1 acquires write lock via BEGIN IMMEDIATE
    db1.exec('BEGIN IMMEDIATE');

    // db2 must fail at BEGIN IMMEDIATE (not silently proceed to write time)
    assert.throws(
      () => db2.exec('BEGIN IMMEDIATE'),
      /database is locked|SQLITE_BUSY/i,
      'second BEGIN IMMEDIATE must fail while first holds the write lock'
    );

    db1.exec('ROLLBACK');
    db1.close();
    db2.close();
  } finally {
    rmSync(dir, { recursive: true, force: true });
  }
});
```

Why Test B is the acceptance gate: if someone changes `tx()` to use plain `BEGIN`, `db2.exec('BEGIN')` would succeed (DEFERRED allows two readers), and the test would fail at `assert.throws` — correctly catching the regression.

## Edge cases

1. **:memory: per-instance**: `new Store(':memory:')` creates a fresh DB per instance — same behavior as better-sqlite3. The `mem()` test helper is correct as-is.

2. **ROLLBACK after auto-rollback**: If a constraint error causes SQLite to auto-abort the statement (but not the transaction), the manual `ROLLBACK` in the catch block correctly ends the transaction. No issue.

3. **StatementSync type export**: If any external consumer imports `StatementSync` from `store.ts` (via re-export), add `export type { StatementSync } from 'node:sqlite'` to `src/index.ts` or add it to the store's import. A grep of the codebase shows no such re-export currently exists — no action needed.

4. **node:sqlite `allowBareNamedParameters`**: The store uses `@name` bindings (not bare `name`). This is supported from Node 22.5.0 onward. No configuration change needed.

5. **WAL mode on :memory: databases**: `PRAGMA journal_mode = WAL` on an in-memory database is silently ignored by SQLite (in-memory databases cannot use WAL). This was the same under better-sqlite3. No concern.
