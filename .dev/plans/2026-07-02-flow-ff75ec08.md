# Plan: declarative exclusive produce-groups (`group:`)

## Goal

Let a step's `produces:` list carry a `group:` entry naming two or more of
that *same step's* singleton sibling stems, plus a commit-exclusivity
contract (`exactlyOne` | `atMostOne` | `atLeastOne`) that the engine enforces
directly. This replaces the prior convention of a router step manually
calling `engine.skip()` on the branch it didn't take (§16.1 routing stays
supported for bespoke logic beyond plain either/or exclusivity — this is a
declarative shortcut for the common case, not a replacement for it).

- **`exactlyOne`** / **`atMostOne`** — once one member goes `green`, the
  engine refuses any commit to a sibling (new `'group-rejected'` outcome,
  same shape as `'schema-rejected'`: value not written, no counters bumped,
  run left open) and auto-skips the untouched siblings in the same
  `settle()` cascade that lands the winner (new `RejectKind: 'exclusive'`).
  The two modes differ only in intent: `atMostOne` also tolerates a producer
  that routes to *neither* member (legal zero-winner end state).
- **`atLeastOne`** — no refusal, no auto-skip; `workflowStatus`'s done-ness
  computation simply stops counting the other members as outstanding once
  any one member is green.

## Files to touch

- `src/types.ts` — `RejectKind` gains `'exclusive'`; new `GroupMode` and
  `GroupDef` types; `StepDef.groups?: GroupDef[]`; `CheckStep['outcome']`
  gains `'group-reject'`.
- `src/defs.ts` — `RawGroup` raw shape; `parseGroup`; `parseProduces` returns
  `{ patterns, groups }` instead of a bare pattern array (every call site
  updated); `buildStep` sets `step.groups`; reject `group:` on a `calls:`
  step's produces and on `generates:` entries; `validateDef` gains
  per-def group validation (unknown `mode`, group declared twice, fewer
  than two members, a member not produced by this step or not a singleton
  produce, a stem double-claimed across groups).
- `src/engine.ts` — new private `groupCasCheck(def, arts, art)`: refuses iff
  a *different* sibling in the same group is already green (`atLeastOne`
  never refuses). Called from all three green-landing branches in
  `Engine.green()`, in the same position relative to other structural
  refusal checks (before CAS/schema — check first, don't mutate on
  refusal):
  1. human-bypass branch (`run === 'human'`)
  2. judge-approve branch (last judge's approve, the actual green-moment
     for a judged produce)
  3. plain (non-judged) producer-commit branch
- `src/model.ts` (the `owenloop check` model checker) — mirror twin of the
  engine change, so the checker explores the same refusal path instead of
  an impossible green:
  - `CascadeOp['skip']` gains an optional `rejectKind` (auto-skip of a
    losing group sibling tags `'exclusive'`; default stays `'structural'`).
  - `maintainDecisions`: auto-skip losing `exactlyOne`/`atMostOne` siblings
    once a winner is green (mirrors the engine's cascade behavior — no
    separate rearm code needed, the existing skipped-artifact rearm loop
    already revives them if the winner is later un-greened).
  - `workflowStatus`: `atLeastOne` members stop counting as outstanding
    once one member is green.
  - `eligibleOutcomes`: new helper `groupWouldReject(def, arts, path)` —
    the checker-side twin of `Engine.groupCasCheck`. Used both in the
    plain-produce branch (model `'group-reject'` instead of `'green'`) and
    in the judges branch (last-judge-approve preempted by `'group-reject'`
    before `'judge-approve'`), matching the engine's ordering.
  - `applyOutcome`: `'group-reject'` is a pure no-op (like
    `'schema-reject'`) — no value applied, no counters moved, just
    `settleInMemory`.
- `test/groups.test.ts` (new) — both-branches-refused, auto-skip, each
  mode's done-ness (including `atMostOne`'s legal zero-winner case),
  cascade re-arm, judges interaction, and a checker/engine differential
  conformance test.
- `test/helpers.ts` — small additions to support the new test file's
  fixture construction.
- `docs/design.md` — new `§26 Declarative exclusive produce-groups
  (group:)` (grammar, the three modes, refusal timing relative to
  schema/CAS, the judges interaction, the model-checker parity guarantee).
- `README.md` — new `### group: — exclusive/inclusive produce groups`
  subsection cross-referencing `docs/design.md` for the full design.

## Key facts from reading the code (do not re-derive, just use)

- `Engine.green()` has three green-landing branches in a fixed order:
  human-bypass (`run === 'human'`), judge-approve (last judge signs off a
  `submitted` stem), and plain producer-commit (no `judges:` declared on
  this produce). All three must run `groupCasCheck` **before** any
  CAS/schema check that could mutate `schemaRejects`/`judgmentRejects` or
  flip `acceptance` — refusal must be side-effect-free, matching how the
  other structural refusal checks are ordered.
- `groupCasCheck` only needs `(def, arts, art)` — it does not depend on the
  CAS fingerprint or the required-inputs list, so it can run before
  `requiredInputs`/`casCheck` are even computed.
- A judged produce's actual "green moment" is the last judge's approve, not
  the producer's initial commit (which lands `submitted`) — so the plain
  producer-commit branch's `groupCasCheck` call is gated on `!hasJudges`
  (a judged produce defers the check to the judge-approve branch).
- `docs/design.md` section numbers: this feature lands as **§26**, not
  §25 — §25 is a pre-existing "model checker scope" section that already
  occupied that slot from an earlier rebase. Every in-code comment and
  the README cross-reference must point at §26.

## Approach (order of work)

1. Types first (`src/types.ts`): `GroupMode`, `GroupDef`, `StepDef.groups`,
   `RejectKind` + `'exclusive'`, `CheckStep['outcome']` + `'group-reject'`.
2. Grammar + validation (`src/defs.ts`): parse `group:` entries alongside
   produce patterns, thread `{ patterns, groups }` through every
   `parseProduces` call site, reject on `calls:`/`generates:`, add
   `validateDef` checks.
3. Engine enforcement (`src/engine.ts`): `groupCasCheck` + wire into all
   three green-landing branches in the correct order (before CAS/schema).
4. Checker parity (`src/model.ts`): `groupWouldReject`, cascade auto-skip,
   done-ness for `atLeastOne`, `'group-reject'` in `eligibleOutcomes` and
   `applyOutcome`.
5. Tests (`test/groups.test.ts`, `test/helpers.ts` tweaks): cover every
   mode, the refusal-ordering guarantee (schema-invalid + already-decided
   group must yield `'group-rejected'`, not `'schema-rejected'`), cascade
   re-arm, judges interaction, and a differential conformance test
   against the live `Engine`.
6. Docs (`docs/design.md` §26, `README.md` group: subsection) — written to
   match the actual implementation, not the other way around.
