### 2b. Modify `buildDef` loop-list parsing (~line 252-255)

Replace:
```typescript
const loops: LoopDef[] = r.loops.map((rl, i) => buildLoop(rl as RawLoop, i));
```

With a two-result split:
```typescript
// A loop-list entry is either a normal loop or an include directive.
// Include directives are collected separately; expansion happens at directory-load
// time once all defs are known (M1-SITE). buildDef returns them in a side-channel
// on the WorkflowDef as `_rawIncludes` so `expandIncludes` can reconstruct position.
//
// Actually: include directives are parsed into a parallel list stored on the def,
// and PLACEHOLDER LoopDef entries are NOT inserted — instead, the raw loop list
// is preserved in an opaque side-channel for expansion. SEE ALTERNATIVE below.
```

> **Design note on placement:** the cleanest approach is to extend `buildDef` to produce a `WorkflowDef` with a `_rawIncludes?: { index: number; directive: ParsedIncludeDirective }[]` private field (omitted from the type, appended at runtime), then have `expandIncludes` read it. However, TypeScript type-safety pushes against undeclared fields.  
>  
> **Chosen approach:** parse the raw loop list in `buildDef` into a union of two shapes — normal loops and include-directive placeholder objects — and thread them through without committing to the `LoopDef[]` type until after expansion. Concretely:
>  
> 1. Define `type RawLoopEntry = { kind: 'loop'; loop: LoopDef } | { kind: 'include'; pos: number; directive: ParsedIncludeDirective }` (internal, not exported).
> 2. Parse each raw entry: if the entry object has an `include` key (duck-check `typeof rl === 'object' && rl !== null && 'include' in rl`), parse it as an include directive; otherwise call `buildLoop` as today.
> 3. `buildDef` returns a `WorkflowDef` whose `loops` array contains only the normal `LoopDef` entries (no placeholders), PLUS a new optional field `_includes?: ParsedInclude[]` listing the directives with their insertion positions.
>  
> Since `WorkflowDef` is a public type and adding undeclared fields breaks the type, the cleanest option is to add an **optional `_includes`** field to `WorkflowDef` in `src/types.ts` marked `@internal`:
>
> ```typescript
> /** @internal Mode 1 include directives before expansion. Not visible to the engine. */
> _includes?: ParsedInclude[];
> ```
>
> **Simpler alternative (recommended):** Do NOT store include directives on `WorkflowDef`. Instead, have `buildDef` receive and return the raw loop list as-is (with include entries present), and have `expandIncludes` receive the **raw YAML loop list** directly — not a `WorkflowDef`. Then the two-phase load pipeline is:
>  1. Parse YAML → raw object (no `buildDef` yet).
>  2. For each raw object, run `extractAndExpandIncludes(rawLoops, resolver)` → fully-expanded raw loops (include entries replaced by prefixed child loops as raw-loop objects).
>  3. Call `buildDef` on the expanded raw object → `WorkflowDef` with no include residue.
>  4. Call `validateDef` on the expanded `WorkflowDef`.
>
> **This is the recommended approach.** It keeps `WorkflowDef` clean, avoids internal fields, and does not require a types.ts change. The expander operates on raw YAML objects before `buildDef`, making it simpler.

### 2b (revised) — split the pipeline before `buildDef`

Instead of modifying `buildDef`, add:

**`expandRawLoops(rawLoops: unknown[], defName: string, resolve: (name: string) => unknown | undefined, stack?: string[]): unknown[]`**

This pure function walks the raw loop list, detects include directives, recursively expands children (operating on their raw YAML objects), prefixes stems, rewires inputs, and returns a flat `unknown[]` that looks like a normal loop list to `buildDef`. Then `buildDef` is called on the expanded raw object unchanged.

However, this approach means the expander works on `unknown` raw YAML, not on typed `WorkflowDef`. It cannot use the parsed child `InputDef[]` to validate `inputs:` map keys without parsing the child first.

**Final chosen approach: type-safe two-phase on `WorkflowDef`**

The cleanest solution that is also type-safe:

1. `buildDef` is modified to skip entries with an `include` key and collect them as `ParsedInclude[]` stored on the `WorkflowDef` in a new optional `_includes` field (added to `WorkflowDef` in `types.ts`, marked `@internal`).
2. `expandIncludes(def, resolve, stack)` reads `def._includes`, expands them, and returns a new `WorkflowDef` with the `_includes` field removed and `loops` containing the fully-spliced expanded list.
3. `loadDefs` and `loadDefsRaw` call `buildDef` then `expandIncludes` then `validateDef`.

**`ParsedInclude` type (internal to `defs.ts`):**
```typescript
interface ParsedInclude {
  pos: number;           // insertion index in the loops array (between loops[pos-1] and loops[pos])
  defName: string;       // the child def name
  as: string;            // the prefix token
  inputs: Record<string, string>; // child-input-name -> outer artifact name (may be empty)
}
```

**`WorkflowDef._includes` (add to `types.ts`):**
```typescript
/**
 * @internal Mode 1 include directives before expansion. Set by buildDef when a
 * loop-list entry has an `include:` key. Consumed and removed by expandIncludes.
 * Never visible to the engine; always undefined on a fully-expanded def.
 */
_includes?: Array<{ pos: number; defName: string; as: string; inputs: Record<string, string> }>;

Add this field to `WorkflowDef` in `src/types.ts` after `dir?: string`.

### 2c. M1-GRAMMAR parsing inside `buildDef` (~line 252, loop-list iteration)

In the `r.loops.map(...)` call, add a guard at the top:

```typescript
const includes: WorkflowDef['_includes'] = [];
const loops: LoopDef[] = [];
let loopPos = 0;
for (const [i, rl] of (r.loops as unknown[]).entries()) {
  if (isIncludeDirective(rl)) {
    // Parse and validate the include directive's own fields (M1-GRAMMAR pre-checks):
    const inc = parseIncludeDirective(rl, i, def.name);
    includes.push({ pos: loops.length, ...inc }); // pos = current end of loops list
  } else {
    loops.push(buildLoop(rl as RawLoop, i));
  }
}
if (includes.length > 0) def._includes = includes;
```

**`isIncludeDirective(v: unknown): boolean`** — duck-check: `typeof v === 'object' && v !== null && 'include' in v`.

**`parseIncludeDirective(raw: unknown, i: number, parentName: string): ParsedInclude`** — M1-GRAMMAR checks:
- `as:` is required: missing → `DefError: loop entry [${i}] include directive missing 'as:'`
- `as:` must be a non-empty string matching `/^[a-z][a-zA-Z0-9_-]*$/`: → `DefError: include 'as:' value '${as}' must be a non-empty identifier matching ^[a-z][a-zA-Z0-9_-]*$`
- `include:` must be a non-empty string: → `DefError: loop entry [${i}] 'include:' must be a workflow name string`
- `inputs:` if present, must be an object mapping strings to strings: → `DefError: include '${as}' inputs: must be an object mapping child input names to outer artifact names`
- Returns `{ defName: string; as: string; inputs: Record<string, string> }`

Post-parse, after the loop, cross-check uniqueness (M1-GRAMMAR). These checks run inside `buildDef` on the collected includes:
- Duplicate `as:` values: → `DefError: include 'as:' value '${as}' is used more than once in workflow '${name}'`
- `as:` colliding with a sibling normal-loop name: collect loop names from the parsed `loops` array, then: → `DefError: include 'as:' value '${as}' collides with loop name '${as}' in workflow '${name}'`

### 2d. Add `expandIncludes` (new exported pure function in `src/defs.ts`)

**Signature:**
```typescript
export function expandIncludes(
  def: WorkflowDef,
  resolve: (name: string) => WorkflowDef | undefined,
  stack?: string[],
): WorkflowDef
```

If `def._includes` is undefined or empty, return `def` unchanged (fast path).

Algorithm (M1-EXPAND):

```
stack = stack ?? [def.name]

Build a slot list from def.loops and def._includes:
  Start with: slots = def.loops.map(l => ({ kind: 'loop', loop: l }))
  For each include in def._includes (sorted by pos ascending):
    insert an include-slot at position pos: ({ kind: 'include', inc })
  Note: positions are insertion indices into the original loops list, so
  process in reverse pos order to avoid index shifts, or rebuild the list
  in a single pass keeping a running index.
```

Simpler: build the output list in order by interleaving loops and includes:
```
result_loops: LoopDef[] = []
result_inputs: InputDef[] = [...def.inputs]
result_outputs: string[] = [...(def.outputs ?? [])]

Flatten: for each slot in the ordered slot list:
  if slot is a normal loop: push to result_loops
  if slot is an include:
    (a) resolve: child = resolve(inc.defName)
        if undefined: throw DefError `include names workflow '${inc.defName}' which does not exist`
    (b) cycle check: if inc.defName is in stack:
        throw DefError `include cycle: ${[...stack, inc.defName].join(' -> ')}`
    (c) recurse: expanded_child = expandIncludes(child, resolve, [...stack, inc.defName])
        (this fully expands any nested includes in child before we splice)
    (d) M1-VALIDATE: inputs map keys must be real child inputs:
        childInputNames = new Set(expanded_child.inputs.map(i => i.name))
        for each key in inc.inputs:
          if not in childInputNames:
            throw DefError `include 'as ${inc.as}' maps input '${key}' which workflow '${inc.defName}' does not declare`
    (e) prefix child loops:
        for each loop L in expanded_child.loops:
          prefixed_loop = prefixLoop(L, inc.as)
          push to result_loops
    (f) handle inputs:
        for each child input I in expanded_child.inputs:
          if inc.inputs[I.name] is set (mapped):
            outer = inc.inputs[I.name]  // outer artifact name, may be input or produce
            rewrite every consume in result_loops whose stem === inc.as + '.' + I.name
              to stem = outer (plain consume to outer; do NOT prefix with inc.as)
            I is NOT added to result_inputs (it becomes an internal consume edge)
          else (unmapped, hoist):
            hoist as new outer input: { ...I, name: inc.as + '.' + I.name }
            push to result_inputs
    (g) merge child outputs:
        for each stem in (expanded_child.outputs ?? []):
          push inc.as + '.' + stem to result_outputs (if not already present)

return new WorkflowDef:
  { ...def, loops: result_loops, inputs: result_inputs, outputs: result_outputs.length > 0 ? result_outputs : def.outputs,
    _includes: undefined  // clear so validateDef sees a clean def }
```

**`prefixLoop(loop: LoopDef, prefix: string): LoopDef`** — internal helper:
- `name`: `${prefix}.${loop.name}`
- `consumes`: for each `ConsumePattern c`: new pattern with `stem = ${prefix}.${c.stem}`, `raw` recomputed, all other fields (mode, binder, suffix) preserved.
  - Reconstruction of `raw`: for plain: `${prefix}.${c.stem}`; for map: `${prefix}.${c.stem}[$${c.binder}]${c.suffix}`; for reduce: `${prefix}.${c.stem}[*]`
- `produces`: for each `ProducePattern p`: new pattern with `stem = ${prefix}.${p.stem}`, `raw` recomputed, kind/binder/suffix/schema preserved.
  - Reconstruction: singleton: `${prefix}.${p.stem}`; collection: `${prefix}.${p.stem}[]`; map: `${prefix}.${p.stem}[$${p.binder}]${p.suffix}`
- `generates`: same transformation as produces, on `loop.generates ?? []`
- `invalidates`: each stem → `${prefix}.${stem}`
- `effect.onInvalidate`: if it's a loop name string (not 'pin'/'escalate'), prefix it: `${prefix}.${oi}`
- All numeric fields (cadenceSecs, parallel, etc.), `body`, `workdir`, `model`, `terminal`, `effect`, `on`, `idleAfter`, `idleAfterMs`: copied as-is
- This is a pure function creating a new `LoopDef` object; does NOT mutate the input.

**Input rewriting after prefixLoop:** the `f` step above rewrites consumes on already-prefixed loops. The consume stem for a mapped input after prefixing is `${inc.as}.${I.name}`. We rewrite any consume in `result_loops` whose `stem` equals this to the outer artifact name. Rewrite means: create a new `ConsumePattern` with `stem = outer, raw = outer, mode = 'plain', suffix = ''`. (Mapped inputs are always plain-consumed; the original child input was a plain consume by definition — inputs are plain artifacts, not collections.)

### 2e. Collection loops — seal and element paths under prefixing

`sealPath(stem)` in `src/paths.ts` returns `${stem}.sealed`. An engine that processes a collection loop uses the `stem` field from the `ProducePattern` to derive the seal path and element paths via `sealPath(stem)` and `elementPath(stem, i)` respectively. Since `prefixLoop` rewrites `p.stem` to `${prefix}.${p.stem}`, the seal path naturally becomes `${prefix}.${p.stem}.sealed` and elements become `${prefix}.${p.stem}[i]`. No change to `paths.ts` is needed — prefixing the `stem` field is sufficient.

The test case (g) in the proposal (collection child) verifies this: a child with a `gather.source[]` collection → after `as: deliver`, stem = `deliver.gather.source`; `sealPath('deliver.gather.source')` = `deliver.gather.source.sealed`. The engine's `engine.ts` uses `sealPath` at runtime and will compute the correct path from the prefixed stem.

### 2f. Two-phase `loadDefs` rewrite

Current `loadDefs` does: build a map of `loadDefFile(file)` calls. `loadDefFile` calls `parseDef` (which is `buildDef` + `validateDef`), so it already throws on errors.

New `loadDefs` (two-phase):

```typescript
export function loadDefs(dir: string): Map<string, WorkflowDef> {
  // Phase 1: build every def's un-expanded form
  const raw = new Map<string, WorkflowDef>();
  // (same filesystem iteration as today, but call buildDef instead of loadDefFile)
  for (const entry of readdirSync(dir)) {
    ... // same stat/dir logic
    // instead of add(loadDefFile(wf)) call:
    const text = readFileSync(wf, 'utf8');
    const def = buildDef(parseYaml(text), basename(wf));
    def.dir = wf;
    if (raw.has(def.name)) throw new DefError(`duplicate workflow name ...`);
    raw.set(def.name, def);
  }

  // Phase 2: expand includes and validate each def
  const out = new Map<string, WorkflowDef>();
  const resolver = (name: string): WorkflowDef | undefined => raw.get(name);
  for (const [name, def] of raw) {
    const expanded = expandIncludes(def, resolver);
    const errors = validateDef(expanded);
    if (errors.length) {
      throw new DefError(`invalid workflow '${name}' (${def.dir ?? 'unknown'}):\n  - ${errors.join('\n  - ')}`);
    }
    out.set(name, expanded);
  }
  return out;
}
```

Note: the resolver passed to `expandIncludes` resolves from the **raw** (un-expanded) map, not the `out` map. `expandIncludes` itself recursively calls `expandIncludes` on the child, so the child gets expanded lazily. This means the resolver should return the un-expanded def; `expandIncludes` handles the recursion. This is correct.

Note 2: `loadDefFile` (for single-file loading, not used in the two-phase flow) continues to call `parseDef` as today. Single-file loading has no resolver so include directives in a single file would fail at expand time (`include names workflow '...' which does not exist`). This is intentional and correct per M1-SITE.

### 2g. Two-phase `loadDefsRaw` rewrite (lint path)

```typescript
export function loadDefsRaw(dir: string): Map<string, WorkflowDef> {
  // Phase 1: build all defs with buildDef (collect errors, skip malformed)
  const raw = new Map<string, WorkflowDef>();
  for (const entry ...) {
    try {
      const def = buildDef(parseYaml(text), basename(full));
      if (raw.has(def.name)) throw new DefError(...);
      raw.set(def.name, def);
    } catch { /* skip malformed */ }
  }

  // Phase 2: expand includes per def (best-effort: skip expansion errors)
  const out = new Map<string, WorkflowDef>();
  const resolver = (name: string): WorkflowDef | undefined => raw.get(name);
  for (const [name, def] of raw) {
    try {
      const expanded = expandIncludes(def, resolver);
      out.set(name, expanded);
    } catch {
      // If expansion fails (e.g. missing child, cycle), still add the un-expanded def
      // so the lint caller can report the include errors via validateDef/lintDef
      out.set(name, def);
    }
  }
  return out;
}
```

Actually, for the lint path the caller (`oweflow lint`) calls `lintDef` on each def. The include pre-checks (missing as:, bad pattern, etc.) throw during `buildDef` and are skipped. The expansion errors (missing child def, cycle) should surface as lint errors. The cleanest approach: catch expansion errors and add them as pseudo-errors to the def's lint output — but since `loadDefsRaw` currently just returns the map and lets the lint command call `lintDef`, it is simpler to:

- If expansion succeeds: put the expanded def in `out` (so `lintDef` runs cross-boundary checks for free).
- If expansion throws a `DefError`: put the UN-expanded def in `out` AND attach the expansion error somehow. Since `loadDefsRaw` cannot return errors today, the simplest: return the un-expanded def — `validateDef` will then fail on the `_includes` field being non-empty... Actually `validateDef` does not know about `_includes`. The un-expanded def has loops with no include entries (since `buildDef` parsed includes into `_includes` and put nothing in `loops` for them). So `validateDef` will not see the expansion error.

**Revised approach for `loadDefsRaw`:** collect expansion errors in a separate map and surface them somehow. But this changes the function signature. The simpler and pragmatically correct approach: on expansion error, throw a `DefError` with the expansion reason — this causes the whole def to be silently skipped by the `catch` in the outer try/catch, meaning the lint user sees nothing for that def. This is acceptable for a lint-path tool (author will also see the error if they try to load with `loadDefs`).

**Final decision:** `loadDefsRaw` wraps the expansion in the same try/catch as the `buildDef` call. If expansion fails, the def is silently skipped. This keeps the existing behavior of `loadDefsRaw` (skip-on-error) consistent.

## 3. Example def: `examples/workflows/full-cycle.yaml`

A concrete working example demonstrating Mode 1:

```yaml
# Mode 1 compile-time include example.
# Includes the delivery workflow under the 'deliver' prefix, maps its seedOwed
# 'proposal' input to the outer 'proposal' artifact, and adds a teardown loop
# that consumes the inlined merge artifact as 'deliver.merge'.
#
# After expansion (one flat WorkflowDef):
#   inputs: [proposal]
#   loops: [provision, deliver.planner, deliver.builder, deliver.reviewer,
#           deliver.merger, teardown]
# The engine sees one graph; cascade crosses the seam for free.
#
# Load with:
#   oweflow --defs examples/workflows defs

name: full-cycle
title: Provision + delivery + teardown (Mode 1 compile-time inline)
description: |
  Embeds the delivery workflow as a compile-time include.
  One flat instance — no engine change.

inputs:
  - name: proposal
    seedOwed: true

loops:
  - name: provision
    consumes: [proposal]
    produces: [environment]
    body: |
      Provision the environment for ${WORKFLOW}. Green `environment`.

  - include: delivery
    as: deliver
    inputs:
      proposal: proposal   # map child's seedOwed input to outer 'proposal'

  - name: teardown
    consumes: [environment, deliver.merge]
    produces: [torn_down]
    generates: [teardown_log]
    body: |
      Tear down the environment. Green `torn_down`.
```

The `delivery` def's `merger` loop has `terminal: true`. After prefixing the loop name becomes `deliver.merger` and `terminal: true` is preserved. The `deliver.merge` artifact is thus terminal in the expanded def. `teardown` consumes it as a plain artifact — valid, because `terminal` does not block consumption, only re-arming.

Note: `deliver.merge` is consumed by `teardown`, so it will NOT trigger a dead-end warning. The `deliver.merger` loop's `terminal: true` suppresses it anyway.

### (e) Nested include (A includes B includes C) fully expands

C: simple two-loop def (c1, c2). B: includes C `as: c`, one own loop. A: includes B `as: b`, one own loop.

### (f) Hard errors — M1-GRAMMAR and M1-VALIDATE pre-expansion checks

Each subcase uses `buildDef` and checks the thrown `DefError` message:

- **(f1) missing `as:`** — include entry with no `as:` field: `DefError: /missing 'as:'/`
- **(f2) bad `as:` pattern** — `as: '123bad'` (starts with digit): `DefError: /must be a non-empty identifier/`
- **(f3) `as:` with leading uppercase** — `as: 'Deliver'`: `DefError` (regex requires `^[a-z]`)
- **(f4) duplicate `as:`** — two include entries both `as: deliver`: `DefError: /used more than once/`
- **(f5) `as:` collides with sibling loop name** — a loop named `deliver` and `include: delivery / as: deliver`: `DefError: /collides with loop name/`
- **(f6) include of non-existent def** — `expandIncludes` call with resolver returning undefined: `DefError: /does not exist/`
- **(f7) inputs map references non-input key** — `inputs: { nonexistent_key: proposal }` where child has no such input: `DefError: /does not declare/`
- **(f8) include cycle** — A includes B, B includes A (set up via two defs and resolver): `DefError: /include cycle:/`

### (g) Expanded def passes validateDef; cross-boundary dangling consume fails with normal error

- Build and expand a valid parent+child; call `validateDef(expanded)`: verify `[]`.
- Build a parent that maps a child input to a non-existent outer artifact (e.g. `inputs: { proposal: 'nonexistent_outer' }`):
  - After expansion, `deliver.planner` consumes `nonexistent_outer`.
  - `validateDef(expanded)` returns an error matching `/nothing produces 'nonexistent_outer'/`.

### `docs/design.md` — add §22

Append a new section at the end of the file:

```markdown
## §22 Mode 1 compile-time workflow composition (`include:`)

A pure `defs.ts` feature — zero engine change. The loader produces an expanded
`WorkflowDef` with the child's loops spliced in, stems prefixed, and inputs
mapped or hoisted. The engine sees one flat graph.

### §22.1 Grammar

```yaml
loops:
  - include: <defName>      # child workflow name
    as: <prefix>            # namespace token; must match ^[a-z][a-zA-Z0-9_-]*$
    inputs:                 # optional: map child seedOwed inputs
      <childInputName>: <outerArtifactName>
```

### §22.2 Expand-then-validate pipeline

1. `buildDef` parses include directives from the loop list into `WorkflowDef._includes`, leaving them out of `loops`.
2. `expandIncludes(def, resolve)` splices the prefixed child loops in place of each directive (M1-EXPAND).
3. `validateDef` runs on the expanded flat def — catching cross-boundary dangling consumes, two-producer conflicts, map/reduce shape errors, and cycles for free.

### §22.3 Prefixing semantics

Every child artifact and loop name is prefixed with `${as}.`:
- Loop name: `planner` → `deliver.planner`
- Produce stem: `plan` → `deliver.plan`
- Consume stem: `plan` → `deliver.plan`
- Collection stem `source[]` → `deliver.source[]` (seal and elements derived correctly from the stem)
- `invalidates` entries prefixed
- `effect.onInvalidate` loop-name strings prefixed

### §22.4 Input rewiring

- **Mapped** (`inputs: { childInput: outerArtifact }`): the child input is not added to the parent's inputs. Every consume referencing `${as}.${childInput}` is rewritten to `outerArtifact`. The rewrite is a plain consume to an existing outer artifact (input or produce); the existing validator checks the reference for free.
- **Unmapped**: the child input is hoisted as `${as}.${childInput}`, preserving `seedOwed`, `producer`, and `schema`.

### §22.5 Recursion and cycle guard

`expandIncludes` maintains an include stack. If a def name appears already on the stack, it throws `DefError: include cycle: <a> -> <b> -> <a>`.

### §22.6 Dev-tooling note (deferred)

Mode 1's name-prefixing (`deliver.plan`, `deliver.merge`) affects `dev` tooling that keys on loop names (worktree wiring, dashboard rendering, fleet shape-matching). Making those prefix-aware is deferred to the dev-tooling PR. Mode 1 v1 is the right tool for **brand-new combined workflows** authored fresh; not for re-skinning an existing delivery line (use Mode 2 for that).
```

```markdown
### Workflow composition (Mode 1 — compile-time `include:`)

A loop-list entry may be an `include:` directive to splice another workflow's loops in at compile time (def-load time). This is pure `defs.ts` — zero engine change. The engine sees one expanded flat graph.

```yaml
name: full-cycle
inputs:
  - name: proposal
    seedOwed: true
loops:
  - name: provision
    consumes: [proposal]
    produces: [environment]
  - include: delivery           # splice delivery's loops in
    as: deliver                 # prefix: deliver.planner, deliver.plan, deliver.merge …
    inputs:
      proposal: proposal        # map child's seedOwed input to outer 'proposal' artifact
  - name: teardown
    consumes: [environment, deliver.merge]   # consume the inlined child artifact directly
    produces: [torn_down]
```

After loading, the expanded def's loops are:
`provision`, `deliver.planner`, `deliver.builder`, `deliver.reviewer`, `deliver.merger`, `teardown` — one flat instance, one cascade, one completion evaluator.

**Grammar rules:**
- `as:` is required; must match `^[a-z][a-zA-Z0-9_-]*$`; must be unique across all includes in the same parent; must not equal any sibling loop name.
- `inputs:` is optional; each key must be a real input of the child def.
- `include:` must name a workflow present in the same def directory.

**Semantics:**
- **Mapped inputs** (`inputs: { childInput: outerArtifact }`) become internal consume edges — `outerArtifact` is an existing artifact in the parent (input or produce), checked by `validateDef`.
- **Unmapped inputs** are hoisted as `${as}.${childInput}` outer inputs, preserving `seedOwed` and schema.
- **Collection loops** in the child prefix correctly: `source[]` → `deliver.source[]`; seal and element paths follow from the prefixed stem.
- **Nested includes** work recursively; cycles are a hard error.

**Constraint:** Mode 1 inlines the child's namespace into the parent. Use it for **brand-new combined workflows** where no downstream tooling expects the old loop names. The existing `delivery` workflow's loops (`planner`, `builder`, `reviewer`, `merger`) become `deliver.planner` etc. — correct for a new `full-cycle` workflow, but would break `dev` tooling that expects the original names. For embedding an existing workflow as an encapsulated unit, use Mode 2 (`calls:`, a later PR).
```
