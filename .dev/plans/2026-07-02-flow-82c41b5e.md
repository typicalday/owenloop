# Plan: teach suffixed reduce (`src[*].child`) in skills/owenloop-author/SKILL.md

## Goal

The engine already supports a suffixed reduce form (`gather.source[*].verdict`,
merged in PR #58, `docs/design.md` §11) and the shipped example workflow
(`examples/workflows/research.yaml`, PR #59) already uses it. The
`owenloop-author` skill — the doc an agent reads to author new workflows — has
not caught up: its teaching example still shows the bare reduce form, its
inline grammar comment and its grammar table omit the suffixed form entirely.
This plan updates skills/owenloop-author/SKILL.md only, at the three touch
points below, so the skill is consistent with docs/design.md §11, README.md's
grammar table, and the shipped research.yaml example. **Docs-only. No engine,
example, or test changes. No other files touched.**

Verified against current `main` (commit `72aadbd`, matches the commit the
proposal cites — no drift): `docs/design.md` §11 is still titled "Collections"
(line 142), and README's grammar table (lines 654-666) is unchanged from what
the proposal describes. Safe to use the line numbers and quoted wording below
directly.

## Source-of-truth wording to mirror

`docs/design.md` §11 (lines 142-161), the two reduce bullets:

```
- **§11.x reduce `src[*]`** — fan-in: see §3.
- **§11.x reduce with suffix `src[*].child`** — fan-in one level deeper: the
  gate is the seal green AND every surviving member's `.child` artifact
  green (not the bare member). Typically fans in over a map step's
  per-element output (`src[$i].child`). Resting inputs / cascade: the
  firing (and the reduce output's fingerprint) rest on the child paths, not
  the bare members — a child rejected or re-greened after the reduce fired
  knocks it back via the ordinary §11.8 forward cascade, no special-case
  machinery. One suffix level only (`src[*].a.b` is a parse error). Bare
  `src[*]` is unchanged (suffix is empty).
```

`README.md` grammar table (lines 656-663), the last two rows (verbatim):

```
| `gather.source[*]` | **reduce** consume | once, when sealed and all surviving members green |
| `gather.source[*].verdict` | **reduce** consume (suffixed) | once, when sealed and every surviving member's `.verdict` is green |
```

`examples/workflows/research.yaml` (current shape, lines 49-58) — the
`synth` step already uses the suffixed form and is the model for touch
point 1's rewritten example:

```yaml
  - name: synth
    # REDUCE: eligible once the collection is sealed and every *surviving*
    # (non-retracted) member's `.verdict` is green. It gates on `check`'s
    # verdicts, not the bare members — so it does NOT fire until every
    # surviving source has actually been checked.
    consumes: ["gather.source[*].verdict"]
    produces: [draft]
    body: |
      Workflow ${WORKFLOW}, run ${RUN}.
      Synthesize the surviving sources into a `draft` answer.
```

## Touch point 1 — the "Collection (fan-out / fan-in)" example

Current text (lines 178-193):

```
**Collection (fan-out / fan-in):**
```yaml
steps:
  - name: gather
    consumes: [question]
    produces: ["gather.source[]"]        # a collection: emit elements, then seal
    body: Find sources; `emit` each, then `seal` when you have enough.
  - name: check
    consumes: ["gather.source[$i]"]      # MAP: one run per element, binds ${INDEX}
    produces: ["gather.source[$i].verdict"]
    parallel: 8                           # fan out — check concurrently
    body: Fact-check this one source; green its `.verdict`, or `retract` it if unusable.
  - name: synth
    consumes: ["gather.source[*]"]        # REDUCE: fires once all survivors are green
    produces: [draft]
    body: Synthesize the surviving sources into a `draft`.
```

Rewrite the `synth` step (and only that step — `gather` and `check` stay as-is)
to:

```
  - name: synth
    consumes: ["gather.source[*].verdict"]  # SUFFIXED REDUCE: fires when every survivor's .verdict is green
    produces: [draft]
    body: Synthesize the surviving sources' verdicts into a `draft`.
```

Notes on the edit:
- `consumes:` changes from `["gather.source[*]"]` to
  `["gather.source[*].verdict"]`, matching the real `research.yaml`.
- Comment text: proposal suggests `# SUFFIXED REDUCE: fires when every
  survivor.s .verdict is green` — the `.s` is a smart-quote/encoding artifact
  in the proposal text for a possessive apostrophe ("survivor's"). Use a plain
  ASCII apostrophe: `# SUFFIXED REDUCE: fires when every survivor's .verdict
  is green`. Do not introduce non-ASCII punctuation.
- `body:` line changes from "Synthesize the surviving sources into a `draft`."
  to something that reflects synthesizing from verdicts, e.g. "Synthesize the
  surviving sources' verdicts into a `draft`." (again, plain apostrophe).
- Formatting: this is a single-line body (unlike research.yaml's multi-line
  `body: |`), matching the existing style of this teaching snippet — keep it
  a one-liner, don't switch to block-scalar style; that would be an
  unrelated formatting change.
- Optional (per proposal, judgment call — recommend including, it's cheap and
  directly asked for): add a one-line note after the code block, in the same
  prose style as the rest of §2c, e.g.:

  > A bare `gather.source[*]` reduce (no suffix) also exists — it gates only
  > on the members themselves, firing as soon as all survivors are green,
  > regardless of any per-element child artifact.

  Place it directly under the closing ` ``` ` of the Collection example,
  before the "**Routing (skip the dead branch):**" section starts (i.e.
  before current line 196). Keep it short — one sentence, consistent with the
  terse voice used elsewhere (e.g. the "A step consumes in exactly one mode…"

## Touch point 3 — the path-grammar table (~lines 245-255)

Current table (lines 245-255):

```
Consume/produce path grammar:

| pattern | meaning |
|---|---|
| `plan` | plain consume / singleton produce |
| `gather.source[]` | collection produce (emit N, then seal) |
| `gather.source[$i]` | map (one run per element; binds `${INDEX}`) |
| `gather.source[$i].verdict` | a map step's per-element output |
| `gather.source[*]` | reduce (fires once, when sealed and all survivors green) |
```

Add one row immediately after the bare-reduce row (`gather.source[*]`), so the
table becomes:

```
| pattern | meaning |
|---|---|
| `plan` | plain consume / singleton produce |
| `gather.source[]` | collection produce (emit N, then seal) |
| `gather.source[$i]` | map (one run per element; binds `${INDEX}`) |
| `gather.source[$i].verdict` | a map step's per-element output |
| `gather.source[*]` | reduce (fires once, when sealed and all survivors green) |
| `gather.source[*].verdict` | suffixed reduce (fires once, when sealed and every survivor's `.verdict` is green) |
```

Wording for the new row is taken from the proposal, with the same apostrophe
fix as touch point 1 (`survivor's`, not `survivor.s`) and mirrors README's
suffixed-reduce row phrasing ("once, when sealed and every surviving member's
`.verdict` is green") closely enough to stay consistent, while keeping this
table's own terser row style (this table already uses shorter phrasing than
README's three-column table — e.g. "reduce (fires once, when sealed and all
survivors green)" vs README's "**reduce** consume | once, when sealed and all
surviving members green"). Keep this table two-column, matching its existing
`| pattern | meaning |` shape — do not switch it to README's three-column
`| pattern | role | fires |` shape; that would be a bigger structural change
than the proposal asks for.

Optional tightening (per proposal, judgment call — recommend including for
clarity): reword the bare-reduce row so the contrast between the two forms is
explicit, e.g.:

```
| `gather.source[*]` | reduce (fires once sealed and all surviving *members* are green) |
| `gather.source[*].verdict` | suffixed reduce (fires once sealed and every surviving member's `.verdict` is green) |
```

The key word to land is the contrast: bare reduce gates on the **members**
(the bare collection elements); suffixed reduce gates on each member's
**child artifact** (`.verdict` here). One suffix level only — do not invent a
`src[*].a.b` example; §11 of design.md explicitly says that's a parse error,
so nothing in the skill should imply nesting works.

## Edge cases / things to get right

- **Apostrophe encoding.** The proposal text contains `survivor.s`,
  `member.s`, `README.md.s` — these are almost certainly smart-quote
  apostrophes (`'`) that got mangled to a literal period+s when the proposal
  was captured as plain text. Do not copy `.s` literally into the skill file.
  Use plain ASCII `'s` throughout, matching the file's existing convention
  (e.g. existing line "A step consumes in exactly one mode only... — the
  loader enforces it" and other possessives elsewhere in SKILL.md use plain
  ASCII apostrophes).
- **Don't touch `gather` or `check` steps** in touch point 1 — only `synth`
  changes. The proposal scopes the rewrite to `synth`.
- **Don't touch §1's prose bullet on collections** (lines 81-84, "A step can
  `emit` N elements then `seal`... a **reduce** step runs once when all
  surviving members are green") — the proposal names exactly three touch
  points (the example, the inline comment, the table) and this bullet isn't
  one of them. Leave it as the simple/plain-language explainer it is; adding
  suffix nuance there would be scope creep.
- **Don't touch the RUN section, `Hard rules / gotchas` section, or anything
  outside §2c/§2d.** Only the three call-outs above.
- **No engine, example (`examples/workflows/*.yaml`), or test files change.**
  `examples/workflows/research.yaml` is reference material to match against,
  not a file to edit.
