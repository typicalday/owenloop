## Domain choice: SLA watchdog for an approval workflow

Domain: a **contract-approval SLA monitor**.

- A contract draft is submitted; it must be reviewed and approved within a deadline.
- If the workflow completes cleanly → the evaluator finalises the record (`cause === 'allGreen'`).
- If the workflow is stuck past the SLA threshold → the evaluator fires, nudges / escalates, re-arms its own alarm, and waits again (`cause === 'idle'`).

This domain is self-explanatory to any reader, naturally motivates the idle/alarm loop, and needs no external tool knowledge.

## Workflow YAML structure

```yaml
name: sla-watchdog
title: Contract approval with SLA watchdog
description: |
  Review and approve a contract within an SLA deadline.
  The monitor loop fires on completion (allGreen) or when stuck past idleAfter (idle).

inputs:
  - name: contract
    seedOwed: true
    schema:
      type: object
      required: [title, submitted_at]
      properties:
        title: { type: string, minLength: 1 }
        submitted_at: { type: string }
      additionalProperties: false

loops:
  - name: review
    consumes: [contract]
    produces: [verdict]
    schema on verdict:
      type: object
      required: [approved, reviewer]
      properties:
        approved: { type: boolean }
        reviewer: { type: string }
      additionalProperties: false
    body: |
      Workflow ${WORKFLOW}, run ${RUN}.
      Read `contract` and produce `verdict` (approved: true/false, reviewer name).

  - name: approve
    consumes: [verdict]
    produces: [approval]
    terminal: true
    body: |
      Workflow ${WORKFLOW}, run ${RUN}.
      The verdict is in. Countersign the contract and green `approval` (terminal).

  - name: monitor
    on: [allGreen, idle]
    idleAfter: 2h
    generates: [outcome]
    body: |
      Workflow ${WORKFLOW}, run ${RUN}.
      Read order.cause:
        allGreen → finalize the record, green `outcome` with status and timestamp.
        idle     → send an escalation nudge; call engine.setAlarm to re-arm in 1 h;
                   green `outcome` with escalation details (engine re-arms it next tick).
```

**Lint-clean invariants this design satisfies:**

| Linter rule | How satisfied |
|---|---|
| `on: []` is hard error | `on: [allGreen, idle]` — non-empty |
| `idle` requires `idleAfter` | `idleAfter: 2h` present |
| dead-end `produces` warning | `outcome` is under `generates:` (lint-exempt) — no consumer needed |
| `outputs:` entries must be produced | `outputs:` not used (monitor uses `generates:`); no dangling entries |
| workflow must have at least one loop | three loops |
| `terminal: true` on `approve` | satisfies intended-sink pattern; no dead-end warning |

**Why `generates:` not `outputs:`?**  The `monitor` loop fires repeatedly (once on allGreen, once per idle re-arm). Its `outcome` is a leaf that no downstream loop consumes. `generates: [outcome]` is the correct declaration for a side-effectful ephemeral output that is intentionally unconsumed (lint-exempt). Using workflow-level `outputs: [outcome]` would also suppress the warning but implies `outcome` is a stable public interface artifact — not appropriate for a repeatedly re-armed evaluator whose value changes each time.

## Edge cases / pitfalls

1. **`generates:` vs `produces:`** — the monitor loop must use `generates: [outcome]`, not `produces: [outcome]`.  Using `produces:` without a downstream consumer triggers a dead-end warning (non-fatal but noisy); using `generates:` suppresses it correctly.
2. **Schema on `contract` input** — include an explicit JSON Schema block so the file is consistent with `onboarding.yaml` (which schemas all inputs); avoids the validator silently accepting garbage at create-time.
3. **Schema on `verdict`** — include a `schema:` block on the `verdict` produces entry to mirror the onboarding style and validate the reviewer's output.
4. **`approval` is terminal** — mark `approve` with `terminal: true`; this satisfies the intended-sink pattern and suppresses the dead-end warning for `approval` (terminal loops are exempt from the dead-end check per `deadEndWarnings` line 980).
5. **`idleAfter: 2h`** — use the string `2h` (matches the README examples); parseDurationSecs will expand it to 7200 s → 7_200_000 ms.
6. **`on:` token order** — write `on: [allGreen, idle]` (allGreen first, as in the README example at line 565); order does not affect engine behavior but matching the README exactly is good pedagogy.
7. **Drive script idle path** — comment out (not uncomment) the idle-path commands in the drive script, because a real reader running the happy path would not hit them.  Make it clear they apply only after `idleAfter` elapses.
8. **No circular reference** — `sla-watchdog` does not `include:` or `calls:` any other workflow; it is fully self-contained.
9. **`loadDefs` resolver** — `provisioned-delivery.yaml` `calls: delivery`; the resolver must find `delivery.yaml` in the same dir.  Adding `sla-watchdog.yaml` does not affect this — it has no `calls:` reference.
10. **File name** — use `sla-watchdog.yaml` (hyphen-separated, all lowercase).  The `name:` field in the YAML must exactly match: `name: sla-watchdog`.  `loadDefs` uses the `name:` field as the map key, not the filename.
