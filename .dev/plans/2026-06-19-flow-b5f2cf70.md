# PR5a Build Plan: Mode 2 `calls:` Grammar + `producedBy` Parent Link

## Goal

Lay the static foundation for Mode 2 runtime workflow composition (`calls:` loops). This PR does NOT implement runtime spawn/cascade-up/outcome-read/re-attach — that is PR5b. It delivers: the `calls:` loop grammar in YAML defs, `validateDef` checks, the cross-def calls-cycle guard, the `producedBy` parent-coordinate link on the workflow store row, `eligibleFirings` exclusion of calls: loops, debt/done correctness, and an example + docs. All design decisions are already settled (M2-*); this plan is execution-level.

### Step 3: `src/defs.ts`

**`RawLoop` interface** (~:44): already has `body?: unknown` as last field; insert `calls?: unknown;` before or after `body`. Also add a `RawCalls` interface for the duck-typed sniffer:
```ts
interface RawCalls {
  calls?: unknown;
  inputs?: unknown;
}
```

**Duck-check function** (analogous to `isIncludeDirective` at ~:149):
```ts
function isCallsDirective(v: unknown): boolean {
  return typeof v === 'object' && v !== null && 'calls' in v && !('include' in v);
}
```
The `!('include' in v)` guard prevents ambiguity, though the YAML schema makes both impossible on the same entry.

**`buildLoop` function** (~:546): at the start, before reading `name`, detect calls:
```ts
if (isCallsDirective(rl)) {
  // parse calls: loop
  const name = asString(rl.name, `loops[${i}].name`);
  const rawCalls = rl as RawCalls;
  const callsTarget = asString(rawCalls.calls, `loop '${name}'.calls`);
  // parse inputs: optional mapping
  const callsInputs: Record<string, string> = {};
  if (rawCalls.inputs !== undefined) {
    if (typeof rawCalls.inputs !== 'object' || rawCalls.inputs === null || Array.isArray(rawCalls.inputs)) {
      throw new DefError(`loop '${name}'.inputs: must be an object mapping child input names to parent artifact names`);
    }
    for (const [k, v] of Object.entries(rawCalls.inputs as Record<string, unknown>)) {
      if (typeof v !== 'string') throw new DefError(`loop '${name}'.inputs: value for key '${k}' must be a string`);
      callsInputs[k] = v;
    }
  }
  // parse produces: (required; exactly one — enforced by validateDef)
  const producesPatterns = parseProduces(rl.produces, `loop '${name}'.produces`);
  const loop: LoopDef = {
    name,
    calls: callsTarget,
    callsInputs,
    consumes: [],          // calls: loops have no consumes (eligibility is engine-managed)
    produces: producesPatterns,
    invalidates: [],
    cadence: DEFAULTS.cadence,
    cadenceSecs: 0,
    maxRunsPerDay: DEFAULTS.maxRunsPerDay,
    parallel: 1,
    maxAttempts: 1,        // never worker-fired; 1 is a safe non-zero sentinel
    maxSchemaFailures: DEFAULTS.maxSchemaFailures,
    workdir: DEFAULTS.workdir,
    body: '',              // machine-handled: no prompt body
  };
  return loop;
}
```
Then the existing `buildLoop` normal path handles non-calls loops.

Actually the cleanest approach is to handle calls: detection INSIDE the loop body of `buildDef` (where `isIncludeDirective` is checked), parallel to how include directives are split out. But the proposal says `buildLoop` sets these fields when `rl.calls` is present. Implement it as a guard at the TOP of `buildLoop`:
```ts
function buildLoop(rl: RawLoop, i: number): LoopDef {
  // M2-GRAMMAR: if this entry carries a 'calls' key, parse it as a calls: loop.
  if (typeof rl.calls !== 'undefined') { ... return callsLoop; }
  // ... normal loop parsing ...
}
```
This keeps the `buildDef` loop-iteration unchanged (both includes and calls are handled inside the per-entry check).

**`validateDef`** (line ~615): add a block after the existing duplicate-name check:
```ts
// M2-VALIDATE: calls: loop rules
for (const l of def.loops) {
  if (!l.calls) continue;
  // (a) called def must exist in the resolution namespace — checked at LOAD time, not here
  //     (validateDef is a pure, per-def function with no resolver; the check lives in loadDefs)
  // (b) exactly one produce
  if (l.produces.length !== 1) {
    errors.push(`calls: loop '${l.name}' must produce exactly one output (got ${l.produces.length})`);
  }
  // (c) no body — already enforced by buildLoop (body is '' for calls: loops)
  // (d) no map/reduce consumes (calls: loops declare no consumes)
  // (e) parent artifact values check: the VALUES of callsInputs must be real parent artifacts
  //     Treat them as consume-stems and re-use the existing producerOf check
  for (const [, parentArtifact] of Object.entries(l.callsInputs ?? {})) {
    if (!producerOf.has(parentArtifact)) {
      errors.push(`calls: loop '${l.name}' maps to parent artifact '${parentArtifact}' which is not produced by any loop or input`);
    }
  }
}
```
**Key decision**: the `calls:` target-def existence check (validation 'a') and the child-input-key check (validation 'b') CANNOT be done in `validateDef` because `validateDef` is a pure per-def function (no resolver). These cross-def checks happen in `loadDefs`/`loadDefsRaw` at Phase 2, similar to how `expandIncludes` validates include input-key mappings. Document this split clearly in a comment.

**Cross-def calls-cycle check** — pure helper added to `defs.ts`, called from `loadDefs` Phase 2 after all defs are expanded. Add a function:
```ts
function detectCallsCycles(defs: Map<string, WorkflowDef>): void {
  // Walk calls: edges; detect back edges using DFS with a grey set.
  // Each node is a def name; each edge is def A has a calls: loop that calls def B.
  const WHITE = 0, GREY = 1, BLACK = 2;
  const color = new Map<string, number>([...defs.keys()].map(k => [k, WHITE]));
  const stack: string[] = [];
  const visit = (name: string): void => {
    color.set(name, GREY);
    stack.push(name);
    const def = defs.get(name);
    const callsEdges = new Set((def?.loops ?? []).filter(l => l.calls).map(l => l.calls!));
    for (const child of callsEdges) {
      if (!defs.has(child)) continue; // missing-def error is reported elsewhere
      const c = color.get(child);
      if (c === GREY) {
        const from = stack.indexOf(child);
        throw new DefError(`calls cycle: ${[...stack.slice(from), child].join(' -> ')}`);
      }
      if (c === WHITE) visit(child);
    }
    stack.pop();
    color.set(name, BLACK);
  };
  for (const name of defs.keys()) if ((color.get(name) ?? 0) === WHITE) visit(name);
}
```
This runs over the FLAT (post-expansion) def map. Include and calls edges are different — this only walks `.calls` fields.

**Cross-def validation in `loadDefs` Phase 2**: after expanding and before the validateDef call for each def, iterate the expanded def's calls: loops to check:
1. The `calls:` target exists in the `raw`/expanded map.
2. Each key in `callsInputs` is a real declared input of the child def.
Then after all per-def validations pass, call `detectCallsCycles(out)` once.

**`loadDefsRaw` Phase 2**: add a best-effort calls-cycle check after expansion (wrapped in try/catch so lint caller sees the error in lint output rather than a throw).

**Re-use of dangling-consume for `callsInputs` values**: the proposal says if we can lean on the existing dangling-consume validator (treating inputs values as consume edges), do so. The cleanest way: in `validateDef`, for each calls: loop, add the `callsInputs` values to the `consumes` set so the existing `producerOf.has(c.stem)` check fires. Do NOT add fake `ConsumePattern` entries to `l.consumes` (that would break eligibleFirings). Instead, add an explicit loop after building `producerOf` to check calls: input wiring values.

### Step 6: `examples/workflows/provisioned-delivery.yaml`

Create a new file. The parent workflow must:
- Have an input (e.g. `proposal`, seedOwed: true)
- Have a `provision` loop (worker) that produces a `sandbox` artifact (consuming `proposal`)
- Have a `deliver` calls: loop that `calls: delivery`, with `inputs: { proposal: proposal }`, producing a `delivered` artifact
- Have a `teardown` worker loop consuming `delivered` (and possibly `sandbox`), producing a terminal artifact
- Load and validate under `loadDefs` alongside the existing `delivery` def

```yaml
name: provisioned-delivery
title: Provision → deliver → teardown (Mode 2 static foundation)
description: |
  Parent workflow demonstrating calls: grammar (PR5a foundation).
  Runtime cascade-up behavior (spawn, green via child outcome) is PR5b.

inputs:
  - name: proposal
    seedOwed: true

loops:
  - name: provision
    consumes: [proposal]
    produces: [sandbox]
    body: |
      Provision a sandbox environment for ${WORKFLOW}.
      Green `sandbox` with the environment details.

  - name: deliver
    calls: delivery
    inputs:
      proposal: proposal
    produces: [delivered]

  - name: teardown
    consumes: [delivered]
    produces: [torn_down]
    terminal: true
    body: |
      Tear down the sandbox. Green `torn_down`.
```

Note: the `sandbox` artifact produced by `provision` is not consumed in v1 (the teardown only needs `delivered`). To avoid a dead-end lint warning, either list `sandbox` under `generates:` in the provision loop or under the workflow's `outputs:`. Use `generates: [sandbox]` or consume it in teardown — the simplest is to have teardown consume both `delivered` and `sandbox`:
```yaml
  - name: teardown
    consumes: [delivered, sandbox]
    produces: [torn_down]
    terminal: true
```
This keeps the wiring clean and avoids lint warnings. `delivered` is the calls: loop's output (a real produced artifact), so the dangling-consume validator will accept it once the calls: produce check passes.

## Edge Cases and Constraints

1. **`calls:` vs `include:` are orthogonal**: `isCallsDirective` checks for `'calls' in v` but NOT `'include' in v` (they are mutually exclusive in practice; the guard prevents confusion).
2. **No `consumes:` on calls: loops**: the `callsInputs` wiring is the conceptual input — but the loop's `LoopDef.consumes` is empty. This means the existing reachability validator in `validateDef` will NOT block a calls: loop even if its parent artifact inputs aren't reachable (intentional: the child def manages its own inputs). However, the `callsInputs` VALUES (parent artifact names) are checked against `producerOf` to catch typos.
3. **Lint dead-end for calls: produce**: the calls: loop's output stem will be consumed by `teardown` in the example. If a def has a calls: loop whose output is NOT consumed, the `deadEndWarnings` function will fire. This is correct — the author should either consume it or list it in `outputs:`. No special case needed.
4. **`pendingOwed` handler-loop exclusion**: calls: loops are NOT handler loops (they don't appear in `effect.onInvalidate`), so `pendingOwed` will correctly seed their output as owed. No special case needed.
5. **Cycle check calls across non-existent defs**: the cycle DFS should skip `calls` targets that don't exist in the def map (the missing-target error is reported separately by loadDefs). Add a `if (!defs.has(child)) continue;` guard.
6. **`validateDef` is pure (no resolver)**: the two cross-def checks (target-def existence, input-key validity) are deferred to loadDefs Phase 2. Document this clearly in the code.
7. **`provisioned-delivery.yaml` must not produce a def-validation error**: the `deliver` calls: loop produces `delivered`; `teardown` consumes `delivered`. The existing dangling-consume check uses `producerOf` which maps `delivered -> deliver` (the calls: loop is treated as a normal producer of its declared output). This should pass without special-casing.
8. **`SCHEMA_VERSION '3'`**: the `migrate()` method runs on every open and is idempotent; existing databases with schema v2 will gain the two new columns on their next open. The SCHEMA_VERSION bump causes `setMeta('schema_version', '3')` to be written, which is purely informational.
9. **`insertWorkflow` signature change**: the second optional param is additive — existing callers (engine.ts) that don't pass it get `undefined`, which means `produced_by_wf` and `produced_by_path` are stored as NULL. No migration needed for existing rows.
10. **Conformance tests (check.test.ts)**: do NOT add any calls: loops to conformance scenarios. The spec says calls: is multi-instance and out of scope for the pure single-instance model twin. Existing 386 conformance tests must pass unchanged.

## Decision Log

- **Where calls-cycle check lives**: `defs.ts` as `detectCallsCycles(defs: Map<string, WorkflowDef>)`, called from `loadDefs` Phase 2 after all defs are expanded and per-def validated. Called once over the full def map. This is the direct analogue of how `expandIncludes` guards include cycles — both are load-time checks. The calls-cycle guard does NOT run inside `expandIncludes` (which only walks include edges).
- **Cross-def validation (target-def existence, input-key validity) lives in `loadDefs` Phase 2**, not in `validateDef` — because `validateDef` is a pure per-def function with no resolver. This matches how `expandIncludes` already validates include input-key mappings.
- **`callsInputs` VALUES are validated against `producerOf`** inside `validateDef` (a per-def check, since we only need to know the parent's own artifacts). The KEY validation (against child inputs) is a cross-def check in `loadDefs`.
- **`consumes:` on calls: loops is empty**: the semantic inputs to a calls: loop are in `callsInputs`, not `LoopDef.consumes`. The existing reachability/dangling-consume validator ignores calls: loops for consume checks. The `eligibleFirings` exclusion makes this moot for scheduling.
- **`body: ''` for calls: loops**: buildLoop sets body to empty string — not undefined — to preserve the invariant that `LoopDef.body` is always a string.
