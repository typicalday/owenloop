# Plan: `bodyFile` step field (load step body from a file)

## Goal

Add a `bodyFile` field to step definitions in `src/defs.ts`. At def-load time,
`bodyFile: relative/path.md` is read from disk and its contents become
`StepDef.body` (same string field the engine already dispatches via
`substitute()` in `src/engine.ts:589`). No engine/dispatch/type changes:
`StepDef.body` stays a plain `string`; only `buildStep`'s construction of it
changes. `body` and `bodyFile` are mutually exclusive per step.

## Files to touch

- `src/defs.ts` — `RawStep` interface, `buildStep`, `buildDef`, and the three
  filesystem loaders (`loadDefFile`, `loadDefs`, `loadDefsRaw`).
- `test/defs.test.ts` — new tests (relative resolution, both-set error, no-base error).
- `skills/owenloop-author/SKILL.md` — grammar reference, `### 2d` block.
- `README.md` — grammar reference, ~line 317 block.
- `docs/design.md` — **skip** (no general step-fields enumeration exists there;
  the only step-grammar block in design.md is §23.1, which is calls:-specific
  and explicitly forbids `body:` on calls: steps — bodyFile doesn't apply there
  either, so nothing to add).

## Key facts from reading the code (do not re-derive, just use)

- `StepDef.body: string` (`src/types.ts:213`) — do NOT add a new field here.
  `bodyFile` resolves *into* this existing field at def-build time.
- `RawStep` (`src/defs.ts:44-65`) currently has `body?: unknown`. Add `bodyFile?: unknown` next to it.
- `buildStep(rl: RawStep, i: number): StepDef` (`src/defs.ts:561`) is where
  `body` is currently set, line 618:
  `body: rl.body === undefined ? '' : asString(rl.body, ...)`.
  This is the ONLY call site that needs new logic (see "Approach" below).
- The `calls:` branch of `buildStep` (`src/defs.ts:562-597`) returns early with

  there; note in a comment that `bodyFile` is a normal-step-only concept if it aids clarity, but not required.
- `buildDef(raw, source)` (`src/defs.ts:304`) calls `buildStep(rl as RawStep, i)` at line 338. This call currently receives no base directory — must thread one through.
- `WorkflowDef.dir` (`src/types.ts:236`) is set by all three loaders AFTER
  `buildDef`/`parseDef` returns (e.g. `src/defs.ts:1063` `def.dir = file`,
  `src/defs.ts:1081` inside `loadDefs`'s `addRaw`, `src/defs.ts:1169`/`1177`
  inside `loadDefsRaw`). **`def.dir` holds the YAML FILE path, not a directory**
  — confirmed by grep, nothing else in the codebase applies `dirname()` to it.
  Do not confuse this with the `baseDir` you are threading into `buildStep`
  (which must be a directory).
- `node:path` currently imports only `{ basename, join }` (`src/defs.ts:24`).
  `dirname` is not imported — add it.
- `readFileSync` is already imported from `node:fs` (`src/defs.ts:23`) — reuse it, don't re-import.
- The only downstream consumer of `body` is `src/engine.ts:589`:
  `prompt: substitute(step.body, { WORKFLOW, RUN, STEP, KEY, INDEX, ... })`.
  This is unaffected — it only ever sees the resolved string. Do not touch engine.ts.
- Existing "mutually exclusive" DefError precedent to match style:
  `src/defs.ts:786`: `` `step '${l.name}': terminal: true and effect: are mutually exclusive; ` + `effect: is the forward spelling — remove terminal: true` ``
  (thrown from `validateDef`, not `buildStep`, for that particular case — but the *message shape* `step '<name>': X and Y are mutually exclusive` is the style to match). For `bodyFile`, the proposal specifies the check should live in `buildStep`/`buildDef` (build time, not validateDef), since it's a per-step shape error like the existing `asString` throws, not a graph-wiring check. Use message:
  `` `step '${name}': set either body or bodyFile, not both` ``.
- Two existing tests assert directly on `.body` and must keep passing unmodified:
  - `test/defs.test.ts:336-360` `'loadDefFile and loadDefs read YAML from disk'` (assertion at line 357, inline `body: |` in a temp-file YAML loaded via `loadDefFile`/`loadDefs`).
  - `test/defs.test.ts:1185-1202` `'parseDef: calls: step sets calls and callsInputs fields'` (assertion at line 1198, `calls:` step with no `body` key, asserts default `''`).
  Neither sets `bodyFile`, so neither is affected by correct implementation — but they're the regression canaries to run first.
- ~30 other inline `body: '...'` / `body: ''` fixtures across `test/defs.test.ts` and `test/include.test.ts` feed `buildDef`/`parseDef`/`validateDef` calls with **no file path** (plain JS objects, not loaded from disk). These calls pass no `baseDir`. Per the proposal's explicit requirement, inline `body:` with no baseDir must keep working exactly as today — this falls out naturally as long as the `baseDir`-less path only errors when `bodyFile` is actually present (never when only `body` is present or neither is present).
- CLI/library call sites confirm a YAML file's directory is always derivable

  call `buildDef`/`parseDef` directly on in-memory objects have no file path —

## Approach

### 1. `RawStep` interface (~line 61, next to `body?: unknown`)

```ts
body?: unknown;
bodyFile?: unknown;
```

### 2. Thread `baseDir` through `buildDef` → `buildStep`

Change signatures:
```ts
export function buildDef(raw: unknown, source?: string, baseDir?: string): WorkflowDef {
  ...
  steps.push(buildStep(rl as RawStep, i, baseDir));
  ...
}

function buildStep(rl: RawStep, i: number, baseDir?: string): StepDef {
```

`baseDir` is the DIRECTORY containing the workflow YAML (already a directory,
not a file — computed by the caller via `dirname()`, see step 4). `buildDef`
itself does no filesystem work; it just forwards `baseDir` down.

Also forward `baseDir` through `parseDef`, since `parseDef` is the public
wrapper most external callers use:
```ts
export function parseDef(raw: unknown, source?: string, baseDir?: string): WorkflowDef {
  const def = buildDef(raw, source, baseDir);
  ...
}
```

### 3. Body/bodyFile resolution inside `buildStep` (replace line 618)

Insert BEFORE constructing the `step` object (need `name` already parsed, which it is at line 599). Replace:
```ts
body: rl.body === undefined ? '' : asString(rl.body, `step '${name}'.body`),
```
with a resolved local variable computed just above the `step` object literal:
```ts
const hasBody = rl.body !== undefined;
const hasBodyFile = rl.bodyFile !== undefined;
if (hasBody && hasBodyFile) {
  throw new DefError(`step '${name}': set either body or bodyFile, not both`);
}
let body: string;
if (hasBodyFile) {
  const bodyFileRel = asString(rl.bodyFile, `step '${name}'.bodyFile`);
  if (baseDir === undefined) {
    throw new DefError(
      `step '${name}': bodyFile requires a workflow loaded from disk (no base directory to resolve '${bodyFileRel}' against)`,
    );
  }
  const resolvedPath = join(baseDir, bodyFileRel);
  try {
    body = readFileSync(resolvedPath, 'utf8');
  } catch (e) {
    throw new DefError(`step '${name}': bodyFile '${bodyFileRel}' could not be read (resolved to '${resolvedPath}'): ${(e as Error).message}`);
  }
} else {
  body = hasBody ? asString(rl.body, `step '${name}'.body`) : '';
}
```
Then in the `step: StepDef = { ... }` literal, use `body,` (the local) instead
of the old inline expression.

Notes:
- `join(baseDir, bodyFileRel)` is correct per requirement 1 (resolve relative
  to the workflow YAML's own directory) — `join` is already imported.
- Wrapping `readFileSync` in try/catch turns a raw ENOENT into a DefError,
  consistent with "DefError for all author-facing errors."
- The `calls:` branch (lines 562-597) returns before this code runs, so
  `bodyFile` on a `calls:` step is inherently ignored/never read — matches
  "No `bodyFile` on `calls:` steps" in Out of scope. (No explicit rejection
  needed since RawCalls duck-typing doesn't look at `bodyFile` either; if you
  want to be defensive you MAY add a check that a `calls:` step with a
  `bodyFile` key throws, but it's not required by the proposal and the two
  branches are already fully separate — skip it to keep the diff minimal
  unless you find it trivial to add symmetrically with how `calls:` steps
  are already forbidden from having a body via convention, not enforcement.)

### 4. Compute and pass `baseDir` from the three loaders

`baseDir` = the DIRECTORY of the YAML file = `dirname(<yaml file path>)`.
Add `dirname` to the `node:path` import (line 24):
```ts
import { basename, dirname, join } from 'node:path';
```

**`loadDefFile(file)`** (~line 1059):
```ts
export function loadDefFile(file: string): WorkflowDef {
  const text = readFileSync(file, 'utf8');
  const raw = parseYaml(text);
  const def = parseDef(raw, basename(file), dirname(file));
  def.dir = file;
  return def;
}
```

**`loadDefs(dir)`** (~line 1076): two call sites build a def from a file —
inside the subdirectory branch (`wf = join(full, 'workflow.yaml')`, ~line 1088)
and the flat-file branch (`full`, ~line 1098). Both currently call
`buildDef(parseYaml(text), basename(...))` inside `addRaw(...)`. Thread the
directory through:
```ts
const addRaw = (def: WorkflowDef, file: string): void => {
  if (raw.has(def.name)) throw new DefError(`duplicate workflow name '${def.name}' under ${dir}`);
  def.dir = file;
  raw.set(def.name, def);
};
for (const entry of readdirSync(dir)) {
  const full = join(dir, entry);
  const st = statSync(full);
  if (st.isDirectory()) {
    const wf = join(full, 'workflow.yaml');
    try {
      if (statSync(wf).isFile()) {
        const text = readFileSync(wf, 'utf8');
        addRaw(buildDef(parseYaml(text), basename(wf), dirname(wf)), wf);
      }
    } catch (e) {
      if (e instanceof DefError) throw e;
      /* no workflow.yaml in this subdir — skip */
    }
  } else if (/\.ya?ml$/.test(entry) && entry !== 'workflow.yaml') {
    const text = readFileSync(full, 'utf8');
    addRaw(buildDef(parseYaml(text), basename(full), dirname(full)), full);
  }
}
```
(`dirname(wf)` and `dirname(full)` both correctly resolve to the directory
containing that YAML file — for the subdirectory case that's the workflow's
own subdirectory, matching requirement 1's example exactly:
`workflows/research/workflow.yaml` + `bodyFile: prompts/research.md` →
`workflows/research/prompts/research.md`.)

Note: `loadDefs`'s Phase 2 also calls `expandIncludes` and `validateDef` on
already-built defs — no baseDir threading needed there since `body`/`bodyFile`
resolution is fully finished in Phase 1 (`buildDef`); `expandIncludes` only
copies/prefixes the already-resolved `body: string` on each `StepDef` (see
`prefixStep`, `src/defs.ts:381-432`, which spreads `...step` and never touches
`body` — confirmed unaffected).

**`loadDefsRaw(dir)`** (~line 1157): same two call sites, same treatment —
add `dirname(wf)` / `dirname(full)` as the third arg to `buildDef`:
```ts
const def = buildDef(parseYaml(text), basename(wf), dirname(wf));
...
const def = buildDef(parseYaml(text), basename(full), dirname(full));
```
`loadDefsRaw` already wraps these in try/catch that silently skips failures
(malformed YAML or bad shape) — a `bodyFile` read failure (bad path, missing
file) will also be caught and skip the def, which is consistent with existing
"lint tool sees only cleanly-parseable defs" behavior. This is a reasonable
and low-risk side effect since `loadDefsRaw` backs `owenloop lint`, a
best-effort surface.

### 5. `buildDef` call sites you must NOT change

`test/defs.test.ts` and `test/include.test.ts` call `buildDef(rawObject)` and
`buildDef(rawObject, source)` directly (no third arg) — these continue to work
unmodified since `baseDir` is optional (`baseDir?: string`) and defaults to
`undefined`, which only matters if a step actually sets `bodyFile`. None of
the existing fixtures do, so all ~30 existing calls behave identically.

## Edge cases

- **Both `body` and `bodyFile` set** → DefError `step '<name>': set either body or bodyFile, not both`. Thrown in `buildStep` before any file I/O, so it fires even if the file doesn't exist or baseDir is missing (ordering matches the "mutually exclusive" check happening first, matching requirement 2's exact wording).
- **`bodyFile` set, no baseDir available** (unit test calling `buildDef`/`parseDef` directly with no third arg) → DefError naming the step and the fact that no base directory is available. Verify this via a "nice to have" test per the proposal.
- **`bodyFile` set, baseDir available, file missing/unreadable** → DefError wrapping the underlying fs error, includes the resolved path for debuggability.
- **Neither `body` nor `bodyFile` set** → `body` defaults to `''`, exactly as today (no behavior change).
- **`calls:` step with `bodyFile`** → structurally unreachable/ignored, since the `calls:` branch returns before the new code runs (matches "No `bodyFile` on `calls:` steps" — out of scope, no explicit rejection needed).
- **`bodyFile` value is not a string** (e.g. a number or list in YAML) → caught by the existing `asString(rl.bodyFile, ...)` coercion helper, consistent with every other field's error style.
- **Absolute vs relative `bodyFile` path**: `join(baseDir, bodyFileRel)` — if an author passes an absolute path, `node:path.join` on POSIX will still concatenate rather than reset to the absolute path (this differs from `path.resolve`). This is fine/intended: the proposal's requirement 1 only asks for relative-to-workflow-dir resolution; nothing asks for absolute-path support, and silently supporting it would undermine the "workflow directory is self-contained/portable" goal. No special-casing needed — just use `join`, do not use `resolve`.
- **`loadDefs`'s two-phase model**: includes (`include:`) copy steps via `prefixStep`, which spreads the already-built `StepDef` — so a child workflow's `bodyFile`-resolved `body` string is correctly inherited into the parent's expanded step list with no re-resolution (and no re-resolution is needed, since resolution already happened against the CHILD's own directory in Phase 1, which is correct — an included child's `bodyFile` should resolve relative to the child's own file, not the parent's).
