# Plan: `runs`, `status.inFlight`, and `reap` CLI verbs

## Goal

Make in-flight work observable and clearable from the CLI, without forcing an
orchestrator to query SQLite directly. Three additive surfaces:

1. `owenloop runs <wf> [--open]` — list runs with claim-state join for open ones.
2. `status <wf>` gains an additive `inFlight` array.
3. `owenloop reap <wf> [--now]` — deliberately run the reaper, with `--now`
   forcing every claim stale (TTL 0) as an admin override for a dead worker.

## Source facts already verified (do not re-derive, just confirm on read)

- `Store.listRuns(workflow)` — `src/store.ts:771` — already exists, used today
  only by `trace` (`src/cli.ts` case `'trace'`).
- `Store.listClaimedTasks()` — `src/store.ts:603` — global (not per-workflow),
  filters `status = 'claimed'`, unused outside tests. **Not** what `runs`
  needs (it's global, not scoped to one workflow) — the join in `runs --open`
  should instead call `Store.listTasks(workflow)` (`src/store.ts:596`, already
  scoped) and match by `(step, key)`, mirroring how `Engine.reap` itself
  iterates (`src/engine.ts:1141`).
- `Engine.reap(workflow, now, def)` — `src/engine.ts:1138-1161` — today returns
  a bare `number` (count reaped). Called from two places: `tick()`
  (`src/engine.ts:440`, destructures into `TickResult.reaped: number`) and
  directly by two engine tests (`test/engine.test.ts:973,997`). Check their
  exact assertions before touching the signature.
- `Engine.status()` — `src/engine.ts:1165-1186` — enriches `debts[]` with
  `failedRuns`/`attempts` from the store but does not expose run ids,
  `claimedAt`, or `heartbeatAt`. `WorkflowStatus` is defined in
  `src/model.ts:758-789` (pure, no store access — the enrichment happens in
  `engine.ts`, same pattern must be used for `inFlight`).
- `isClaimFresh` (`src/engine.ts:531-536`) and `effectiveTtl`
  (`src/engine.ts:522-524`) are the two private helpers that encode "is this
  claim still alive" — `reap --now` must reuse these semantics (TTL 0), not
  duplicate the staleness math.
- CLI dispatch is a big `switch` in `dispatch()` in `src/cli.ts:346-640`; each
  case is a thin adapter — parse args, call one engine/store method, `print()`
  the JSON, return an exit code. `USAGE` is a template string at
  `src/cli.ts:179-208`; README's `## CLI reference` table
  (`README.md:201-224`) mirrors it and must be updated too. There's also a
  duplicate command list in the top-of-file doc comment (`src/cli.ts:10-27`)
  that needs the same additions.
- The "stale lease" commit-time guard is `Engine.openRun`
  (`src/engine.ts:1597-1606`): throws `` `run ${run} no longer holds its lease
  (reaped or superseded)` `` whenever the task's current claim no longer points
  at that run id. `reap --now` reaping a claim and re-arming its task is
  exactly what later makes a stale run's `green`/`close` hit this path — no
  new invalidation code needed, just confirm behavior with a test.
- Test conventions: `node:test` + `node:assert/strict`, tests live flat under
  `test/`, `test/helpers.ts` provides `def`/`step`/`input`/`arts` builders,
  `test/cli.test.ts` drives the CLI in-process via `main(argv, io)` with an
  injected `CliIO` (preferred over spawning a subprocess for new coverage),
  `test/heartbeat.test.ts` is the closest existing analog (claim/TTL/reap
  scenarios) — follow its `makeEngine` harness shape for new engine-level reap
  tests. CI gate is `bash .dev/checks.sh` → `npm run check`.

## Step 1 — extend `Engine.reap` to report what it reaped, and add a TTL override

File: `src/engine.ts`.

`reap()` currently returns `number`. Two callers exist: `tick()` (wants the
count) and two direct test calls in `test/engine.test.ts:973,997` (check their
assertions on read). `TickResult.reaped` (`src/engine.ts:98`) is a documented
public field typed `number` — do not change its type.

- Add a new private helper `reapDetailed(workflow, now, def, opts?):
  { count: number; details: Array<{ step: string; key: string; run?: string }>
  }` containing the existing loop body from `reap()`, but instead of just
  `n++`, push `{ step: task.step, key: task.key, run: task.run }` before
  re-arming.
- Add an optional `ttlOverride?: number` to the options bag so `reap --now`
  can pass `0`. Thread it through: `const ttl = opts.ttlOverride ?? this.effectiveTtl(stepDef);`.
  Prefer this explicit override over faking the clock (`now`), since faking
  `now` would also perturb any future cadence/budget math if this helper were
  ever reused elsewhere — keep the two concerns decoupled.
- Keep the existing public `reap(workflow, now, def)` signature and behavior
  exactly as-is (delegates to `reapDetailed` and returns `.count`) — this
  preserves `tick()`'s `reaped: number` contract and the two existing
  `test/engine.test.ts` call sites verbatim.
- Add a new public method `reapWithDetails(workflow, now, def, opts?:
  { ttlOverride?: number }): { count: number; details: Array<{ step: string;
  key: string; run?: string }> }` that the CLI's new `reap` verb calls
  directly (bypassing `tick`'s full maintain/eligibility/claim cycle — the
  proposal's verb is "run the reaper deliberately," not a full tick). Without
  `--now` it applies the same TTL rules a tick would (no `ttlOverride`, so
  `effectiveTtl`'s per-step-or-engine-default logic applies normally). Do NOT
  call `maintainCalls` from this path — `reap` is scoped to lease cleanup
  only; touching `calls:` child maintenance would broaden the change's blast
  radius beyond what the proposal asks for.
- `reapWithDetails` runs inside `this.store.tx(...)` (same transactional
  discipline as every other mutating engine method) since it writes task rows
  via `store.putTask`.

## Step 2 — CLI verb `reap <wf> [--now]`

File: `src/cli.ts`.

- New case in the `switch`, grouped near `tick` (conceptually adjacent):
  ```ts
  case 'reap': {
    const wf = need(args, 1, 'workflow');
    const nowFlag = flag(args, 'now');
    const wfRow = store.getWorkflow(wf);
    if (!wfRow) throw new CliError(`workflow not found: ${wf}`);
    const def = ctx.defs.get(wfRow.def);
    if (!def) throw new CliError(`unknown workflow definition '${wfRow.def}' (looked in ${ctx.defsDir})`);
    const result = engine.reapWithDetails(wf, nowMs(), def, nowFlag ? { ttlOverride: 0 } : {});
    print(io, { reaped: result.count, details: result.details });
    return 0;
  }
  ```
  (Import `nowMs` from `./util.ts` in `cli.ts` if not already imported. The
  `wfRow → ctx.defs.get(wfRow.def)` def-resolution pattern mirrors the
  existing `trace` and `graph` cases, including their exact `CliError`
  messages, for consistency.)
- Print shape per the proposal: `{ reaped, details: [{step, key, run}] }` —
  `reaped` is the count (matches `TickResult.reaped`'s naming for vocabulary
  consistency across the CLI).
- Update `USAGE` (`src/cli.ts:179-208`) — add near `tick`:
  ```
  reap <wf> [--now]                      run the reaper; --now forces every claim stale (TTL 0)
  ```
- Update the doc comment block at the top of `cli.ts` (lines 10-27) the same way.

## Step 3 — CLI verb `runs <wf> [--open]`

File: `src/cli.ts`.

- New case, positioned near `trace`/`list` (same observability family):
  ```ts
  case 'runs': {
    const wf = need(args, 1, 'workflow');
    const openOnly = flag(args, 'open');
    const runs = store.listRuns(wf);       // src/store.ts:771, already wf-scoped
    const tasks = store.listTasks(wf);     // src/store.ts:596, for the claim join
    const now = nowMs();
    const taskByKey = new Map(tasks.map((t) => [`${t.step}${t.key}`, t]));

    const rows = runs
      .filter((r) => !openOnly || r.outcome === undefined)
      .map((r) => {
        const base = {
          run: r.id,
          step: r.step,
          key: r.key,
          outcome: r.outcome ?? 'open',
          createdAt: r.createdAt,
          updatedAt: r.updatedAt,
        };
        if (r.outcome !== undefined) return base; // only join claim state for OPEN runs
        const task = taskByKey.get(`${r.step}${r.key}`);
        if (!task || task.run !== r.id) return base; // superseded/reaped — no live claim to join
        return {
          ...base,
          claimedAt: task.claimedAt,
          heartbeatAt: task.heartbeatAt,
          attempts: task.attempts,
          claimAgeMs: task.claimedAt !== undefined ? now - task.claimedAt : undefined,
          heartbeatAgeMs: task.heartbeatAt !== undefined ? now - task.heartbeatAt : undefined,
        };
      });
    print(io, rows);
    return 0;
  }
  ```
  Notes / edge cases:
  - Use a ``-joined composite key (or a small tuple map) for the
    `(step, key)` join, not plain string concatenation with a space — a step
    or key value could itself contain a space; this mirrors the deterministic
    id style already used by `taskId`/`artifactId` in `store.ts` (which use
    `detId` with explicit separators) rather than inventing a fragile ad hoc
    join key.
  - `store.listRuns` returns every run ever created for the workflow;
    filtering by `--open` happens in the CLI layer, matching how `status
    --all`'s filter logic already lives in the CLI (`src/cli.ts:391-404` is
    the precedent for CLI-side filtering of a store-wide read).
  - "Open" = `r.outcome === undefined` — the same predicate `openRun` uses
    (`src/engine.ts:1600`).
  - Confirm on read whether `RunRow.key`/`TaskRow.key` are ever `undefined`
    vs. always a plain string (schema says `NOT NULL DEFAULT ''` for both,
    and `mapRun`/`mapTask` read them directly with no `??` fallback) — if so,
    no null-coalescing is needed on the join key, only on the optional
    `claimedAt`/`heartbeatAt`/`run` fields which genuinely are optional.
  - Only join claim state for OPEN runs, per the proposal ("for OPEN runs
    join the owning task's claim state"). A closed run's row never gets claim
    fields, even if a `(step, key)` was later reclaimed by a different run —
    guard on `task.run === r.id`, not just presence of a task, so a closed
    run never shows a newer/unrelated run's live claim data.
  - `attempts` on the task row is the same reaper-strike counter
    `status.debts[].attempts` surfaces (`src/engine.ts:1182-1184`) — reuse
    directly, do not recompute.
  - JSON output only; no text-format flag requested by the proposal, so don't
    add one speculatively.
- `--open` via the existing `flag(args, 'open')` helper (`src/cli.ts:95-98`),
  same as `flag(args, 'all')`/`flag(args, 'terminal')` — no new parsing
  primitive needed.
- Update `USAGE` and the top-of-file doc comment, same as Step 2:
  ```
  runs <wf> [--open]                     list this instance's runs (+ claim state for open ones)
  ```

## Step 4 — `status` gains additive `inFlight`

Files: `src/engine.ts` (`status()`, lines 1165-1186), `src/model.ts`
(`WorkflowStatus`, lines 758-789).

Decision: **`inFlight` is engine-only enrichment**, not derived by the pure
`model.ts#workflowStatus`, for the same reason `failedRuns`/`attempts` on
debts are enrichments applied in `engine.ts` after calling the pure function
(`src/engine.ts:1174-1184`) — `model.ts` has no store access and must stay
pure/testable in isolation. So:

- Add `inFlight` to the `WorkflowStatus` interface in `src/model.ts:758-789`
  as a non-optional array field (type it as always-present, and have
  `workflowStatus()` return `inFlight: []` always) — keeps the type
  single-source-of-truth in model.ts (consistent with `pending`/`debts`/
  `blocked` all being declared there), while the actual population happens in
  `engine.ts#status()`, exactly mirroring the existing `debts[].failedRuns`/
  `.attempts` enrichment pattern. This is additive to the type (a new named
  field, not a change to any existing key), so `dev fleet`-style consumers
  parsing named fields are unaffected.
- Shape (per proposal): for each **claimed** task in the workflow —
  `{ step, key, run, claimedAt, heartbeatAt, attempts, claimAgeMs,
  heartbeatAgeMs }`.
- Implementation in `Engine.status()` (`src/engine.ts:1165-1186`), after the
  existing debts-enrichment loop:
  ```ts
  const now = nowMs();
  const inFlight: WorkflowStatus['inFlight'] = [];
  for (const task of this.store.listTasks(workflow)) {
    if (task.status !== 'claimed') continue;
    inFlight.push({
      step: task.step,
      key: task.key,
      run: task.run,
      claimedAt: task.claimedAt,
      heartbeatAt: task.heartbeatAt,
      attempts: task.attempts,
      claimAgeMs: task.claimedAt !== undefined ? now - task.claimedAt : undefined,
      heartbeatAgeMs: task.heartbeatAt !== undefined ? now - task.heartbeatAt : undefined,
    });
  }
  st.inFlight = inFlight;
  ```
  `status()` has no `now` parameter today and none should be added — it's a
  derived read with a zero-clock-parameter public signature used throughout
  the codebase and by embedders (see `docs/embedding.md`'s `engine.status(wf)`
  example) — introducing `nowMs()` internally is consistent with how `reap`'s
  own default parameter (`now = nowMs()`) and other read paths already
  default the clock.
- This does not touch `st.done`, `st.debts`, `st.eligible`, `st.blocked`, or
  `st.pending` — purely additive. Before implementing, search `test/*.test.ts`
  for `deepEqual`/`deepStrictEqual` against a full `WorkflowStatus` object —
  any such test needs `inFlight` added to its expected value. The acceptance
  bar explicitly permits this ("existing tests untouched except where they
  assert exact status shape").

## Step 5 — tests

New tests, following existing conventions (`node:test`, `node:assert/strict`,
in-memory store via `openStore(':memory:')`, `test/helpers.ts` builders,
`test/cli.test.ts`'s `makeCli()` harness for CLI-level tests since these are
new CLI verbs). Prefer CLI-level tests for `runs`/`reap` (pure CLI surface,
easiest to assert JSON shape end-to-end); engine-level (`makeEngine` pattern
from `test/heartbeat.test.ts`) is fine for `reapWithDetails`/`inFlight` if
that reads more naturally once writing the tests.

1. **`runs` listing happy path**: create an instance, tick to claim a run,
   `green` + `close` it, tick again to produce a second run; call
   `runs <wf>` and assert both rows appear with correct
   `outcome`/`step`/`key`, and the closed run has no claim-state fields.
2. **`runs --open` + claim join**: tick to claim a run (leave it open, do not
   green/close), call `runs <wf> --open`; assert exactly the open run is
   returned with `claimedAt` present, `attempts` present, and `claimAgeMs
   >= 0`.
3. **`status.inFlight` presence and shape**: tick to claim a run, call
   `status <wf>` (CLI or `engine.status` directly), assert `inFlight` has one
   entry with all seven fields and `run` equal to the claimed run id; then
   close the run and assert `inFlight` is empty again.
4. **`reap --now` clears a synthetic stale claim and invalidates the old
   run**: tick to claim run R1 with a normal (long/default) TTL so it would
   NOT be reaped by a plain `tick`/`reap` at a near-future `now`; call
   `reap <wf> --now`; assert the CLI reports `reaped: 1` with `details`
   containing `{step, key, run: R1}`; then attempt `green <wf> R1 <path>
   --value '{}'` and assert it fails with the `no longer holds its lease`
   error (matching stderr/exit code) — this is the "designed stand-down
   signal" the proposal calls out explicitly, so assert on the actual error,
   not just that reap ran.
5. **`reap` without `--now` leaves a fresh claim alone**: tick to claim a run
   at `now=0` with the default (long) TTL; immediately call `reap <wf>`
   (no `--now`) at the same/nearby `now`; assert `reaped: 0` and that a
   subsequent `green`/`close` on that run still succeeds.

## Assumptions made (state these in the PR description)

1. `reap`'s existing public `number`-returning signature and behavior are
   preserved byte-for-byte (new `reapWithDetails` method added alongside it)
   to avoid touching `TickResult.reaped`'s documented type or the two direct
   `test/engine.test.ts` call sites. This is a contained, reversible choice,
   made rather than escalated per the protocol.
2. `reap <wf>` (no flags) on a workflow with no stale claims returns
   `reaped: 0, details: []` — a normal no-op admin action, not an error
   (mirrors `tick`'s reap being silently 0 most of the time).
3. `runs`'s claim-state join fields (`claimedAt`, `heartbeatAt`, `attempts`,
   `claimAgeMs`, `heartbeatAgeMs`) are omitted entirely (not null-filled) on
   rows where they don't apply (closed runs, or an open run whose task was
   superseded/reaped between reads) rather than emitting `null` — consistent
   with how `mapRun`/`mapTask` in `store.ts` omit absent optional fields
   rather than nulling them.
4. `inFlight` is computed with `nowMs()` read once per `status()` call, not
   threaded as a parameter — consistent with `status(workflow)`'s existing
   zero-clock-parameter public signature; embedders needing a fixed clock for
   deterministic tests already control time via `Engine`'s constructor-level
   `reapTtlMs` and direct store manipulation, not via `status()`.
5. `reap <wf>` (no flags) with no explicit TTL override runs the same
   per-step `effectiveTtl` logic `tick()` already uses — "the same TTL rules
   a tick would apply," per the proposal — so no new TTL configuration
   surface is introduced.
