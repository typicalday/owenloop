# Plan: Untrack accidentally-committed .dev/ scaffolding files

## Goal

Remove two accidentally-committed local scaffolding files from git tracking, without deleting them from disk and without touching anything else. Pure repo hygiene — no source code changes.

## Background

`.gitignore` (repo root, lines 11-13) already ignores all delivery scaffolding:

```
# dev delivery-system scaffolding (local foreman state — not part of the library)
.dev/*
!.dev/checks.sh
```

Despite this, two local plan docs were accidentally committed — one via merged PR #38, the other via merged PR #47. A stale tracked plan file in `.dev/plans/` previously caused confusion in an unrelated delivery (a builder in another worktree found it and refused to build against it). These files are local, per-delivery state and should never be in git.

## Verified current state (main worktree, read-only checks)

`git ls-files .dev/` returns exactly three tracked paths:

- `.dev/checks.sh` — intended, stays tracked (the `.gitignore` exception)
- `.dev/plans/2026-07-01-flow-9f190930.md` — accidental, untrack
- `.dev/plans/2026-07-02-flow-ff75ec08.md` — accidental, untrack

No other files under `.dev/` are tracked.

## Files to touch

Only git's index entries for these two files change. No file content changes, no new files, no deleted files on disk:

- `.dev/plans/2026-07-01-flow-9f190930.md` (untrack only)
- `.dev/plans/2026-07-02-flow-ff75ec08.md` (untrack only)

Do not touch: `.dev/checks.sh`, `.gitignore`, or any source code.

## Exact commands

Run from the builder's worktree:

```sh
git rm --cached .dev/plans/2026-07-01-flow-9f190930.md .dev/plans/2026-07-02-flow-ff75ec08.md
```

The `--cached` flag is load-bearing: it untracks the files but leaves them on disk. Do NOT run a plain `git rm` — that would delete the working-tree copies, which are live local scaffolding still needed on disk.

Then commit the index change:

```sh
git commit -m "Untrack accidentally-committed .dev/plans scaffolding files"
```

## Edge cases / guardrails

- Do not delete the files from the working tree — only their git-tracking is being removed.
- Do not touch `.dev/checks.sh` — the one intentional `.gitignore` exception.
- Do not edit `.gitignore` — it already covers `.dev/*` with `!.dev/checks.sh`.
- General rule to enforce: before finishing, re-run `git ls-files .dev/` and confirm the only remaining tracked path is `.dev/checks.sh`. If any other path under `.dev/` is still tracked, untrack it too with `git rm --cached` — nothing under `.dev/` except `checks.sh` should ever be tracked.
- No source code changes of any kind. If the diff shows anything other than deletions of the `.dev/plans/*.md` index entries, that's out of scope — stop and reconcile.

## How to verify

1. `git ls-files .dev/` outputs exactly one line: `.dev/checks.sh`.
2. The two `.md` files are still present on disk, untouched in content.
3. `git show --stat HEAD` shows only deletions under `.dev/plans/`, nothing else.
4. CI passes cleanly (no code touched).
