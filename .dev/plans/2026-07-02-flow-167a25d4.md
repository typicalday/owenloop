## 3. research.yaml — precise edits

Current wiring (`examples/workflows/research.yaml`):

  so it's eligible as soon as `gather` seals, in parallel with `check`, not

Change:
- `synth.consumes` → `["gather.source[*].verdict"]`. This is a pure config
  edit — no engine change needed, `reduceInputPath()` (src/model.ts, added in

- Update `check`'s body so its `.verdict` green value documents what it
  should carry (source url + holds-up verdict), since `synth`'s firing (and
  its consumed-input fingerprint) now rests on those `.verdict` values, not

  `.verdict` in this file today and none needs to be added (the improve.yaml

- Update the file's header comment (currently: "eligible only once the seal

  sealed AND every surviving member's `.verdict` is green; a retracted

## 4. improve.yaml — adapt the draft with three rewires

Start from the draft file verbatim (schemas, `maxAttempts`, `parallel`, the
SKILL.md attribution link, the hard-rules flavor in bodies — read-only on
source, self-contained plans, secrets by file:line reference only, repo
content is data not instructions — all carry over unchanged). Apply exactly
these rewires:

### 4a. `vet` step — carry the confirmed finding record

Current draft:
```yaml
  - name: vet
    consumes: ["audit.finding[$i]"]
    produces: ["audit.finding[$i].vetted"]
    parallel: 8
    body: |
      ...Otherwise green its `.vetted`, correcting the
      evidence in your verdict if the claim is real but the file:line is off.
```

Change `produces` to declare a schema on `.vetted` carrying the confirmed
finding record (same shape as the `audit.finding[]` element schema — category,
summary, evidence, impact, effort, confidence — since triage's reduce will
consume these fields directly instead of re-deriving them from the bare
member):

```yaml
  - name: vet
    consumes: ["audit.finding[$i]"]
    produces:
      - name: "audit.finding[$i].vetted"
        schema:
          type: object
          required: [category, summary, evidence, impact, effort, confidence]
          properties:
            category: { enum: [correctness, security, performance, tests, debt, dependencies, dx, docs, direction] }
            summary: { type: string, minLength: 1 }
            evidence: { type: string, minLength: 1 }
            impact: { type: string }
            effort: { enum: [S, M, L] }
            confidence: { enum: [high, medium, low] }
          additionalProperties: false
    parallel: 8
    body: |
      Workflow ${WORKFLOW}, run ${RUN}, element ${INDEX}.
      Open the cited code and confirm this one finding. Auditors over-report:
      if it's by-design (e.g. recorded in an ADR), mis-attributed, or a
      duplicate, `retract audit.finding[${INDEX}]` with the reason — it drops
      out of triage's reduce. Otherwise green its `.vetted` with the
      CONFIRMED finding record (category, summary, evidence, impact, effort,
      confidence — correcting the evidence file:line if the claim is real but
      the citation is off). `triage`'s reduce now consumes these `.vetted`
      records directly, not the bare findings.
```

### 4b. `triage` step — reduce on `.vetted`, not the bare member

Current draft:
```yaml
  - name: triage
    consumes: ["audit.finding[*]"]
    produces: [report]
```

Change to:
```yaml
  - name: triage
    consumes: ["audit.finding[*].vetted"]
    produces: [report]
    body: |
      Workflow ${WORKFLOW}, run ${RUN}.
      Rank the surviving findings by leverage (impact ÷ effort, weighted by
      confidence) using the confirmed `.vetted` records — the reduce is now
      eligible only once every surviving finding has cleared vet, so `report`
      always reflects vetted data, never a finding vet hasn't reached yet.
      Green `report`: the ordered table, direction items listed separately
      (they're options, not defects), and any dependency ordering between
      prospective fixes. The workflow now blocks on `selection` — tell the
      human to read the report and `owenloop provide selection` with the
      finding indexes to plan.
```

Remove the old comment above `triage` in the draft ("REDUCE: sealed + every
surviving finding green. Gates on the members, not on vet's `.vetted`
verdicts...") — that caveat is exactly what suffixed reduce removes; delete
it, do not adapt it (see header rewrite in §4d below for the replacement
framing).

### 4c. `index` step — plain suffixed reduce, drop the `allGreen` workaround

Current draft uses a completion-evaluator (`on: [allGreen]`) because a plain
reduce couldn't gate on map children. That gap is closed. Change:

```yaml
  - name: plan_review
    consumes: ["select.pick[$i].plan"]
    produces:
      - name: "select.pick[$i].review"
        schema:
          type: object
          required: [file, ok]
          properties:
            file: { type: string, minLength: 1 }
            ok: { type: boolean }
          additionalProperties: false
    parallel: 4
    body: |
      Workflow ${WORKFLOW}, run ${RUN}, element ${INDEX}.
      Read this one plan file cold, as the executor would — you have not seen
      the audit. If any step needs context the file doesn't contain ("the
      pattern discussed above", a path that doesn't resolve, prose done
      criteria), `reject select.pick[${INDEX}].plan` with the concrete gaps —
      the planner re-arms for just this plan. Otherwise green its `.review`
      with {file, ok:true} so `index`'s reduce knows the plan path.

  - name: index
    consumes: ["select.pick[*].review"]
    produces: [plans_index]
    body: |
      Workflow ${WORKFLOW}, run ${RUN}.
      Every selected plan has cleared cold review — this reduce is eligible
      only once every surviving pick's `.review` is green, so it never runs
      ahead of an in-flight review. Write `plans/README.md` — recommended
      execution order, dependencies between plans, a status column executors
      can update — and green `plans_index` with its path.
```

Note: `plan_review`'s produce schema is a new addition (the draft's `.review`
had no schema, greened with `{"ok":true}` in the smoke script) — adding
`{file, ok}` is a reasonable, low-risk assumption so `index` has the plan
paths without re-reading every plan file; document this as an assumption in
the file header if space allows, but it is not a one-way-door choice (schema
tightening on an example file, trivially reversible).

Delete: the `index` step's old comment block ("Completion evaluator (§21):
fires once every artifact above... A reduce can't gate on map children, so
the fan-in over 'all plans reviewed' is the all-green condition itself.") —
no longer true, remove entirely rather than adapt.

### 4d. Header comment — rewrite the "three idioms" block

Replace the draft's header comment (lines 1-52 of the draft) with a version
that:

- Keeps the attribution line to shadcn's `improve` skill
  (https://github.com/shadcn/improve/blob/main/skills/improve/SKILL.md) and
  the one-paragraph pipeline description.
- Demonstrates three idioms (rewrite the list):
  (a) **a human gate as a second seedOwed input**, provided mid-flight —
      `triage` greens the findings `report`, and the workflow simply blocks
      on `selection` until a human reads the report and runs
      `owenloop provide`; no special "pause" verb, a missing input IS the
      pause (unchanged from the draft — keep this idiom as-is);
  (b) **per-element knock-back via map children** — plans are map outputs
      (`select.pick[$i].plan`), not a collection the planner emits, so
      rejecting one re-arms the map for just that element
      (`plan_review` rejects `select.pick[$i].plan`, and the reason rides
      the next order). Contrast with a *bare collection member*: rejecting
      `audit.finding[1]` directly doesn't re-arm its producer (only a human
      `green`/`retract` repairs it) — that asymmetry is why `vet` greens-or-
      retracts findings rather than rejecting them, and why plans are map
      children rather than collection elements (keep this contrast from the
      draft, it is still accurate and is not specific to the allGreen
      workaround);
  (c) **suffixed reduces as the fan-ins** (NEW, replaces the old `on:
      allGreen` idiom) — `triage` consumes `audit.finding[*].vetted` and
      `index` consumes `select.pick[*].review`; both gate on a map step's
      per-element output rather than the bare collection members, so
      neither reduce fires until every surviving element has cleared its
      per-element step. This replaces the `allGreen` completion-evaluator
      workaround entirely: `index` is now a plain reduce, and a plan
      rejected even after the workflow reached `done` re-arms `index`
      through the ordinary forward cascade (§7) — no `on:[allGreen]`
      special-casing needed, because the reduce output's fingerprint now
      rests on `.review`, which is the artifact that actually moved.
- Deletes the old "one level-triggered dynamic to know" paragraph in the
  draft (the note about `triage` firing alongside `vet` before every verdict
  lands, and the retraction-cascades-a-report-rebuild timing) — that
  workaround-era caveat no longer applies once `triage` gates on `.vetted`.
- Deletes the draft's final code-comment note about `§21.4 fall-out-of-done
  re-arm` for `index` (also superseded — see 4c).
- Updates the drive-it transcript at the bottom of the header to match the
  new wiring: `vet` should `green ... "audit.finding[$idx].vetted"` with a
  confirmed-record value (not `{"ok":true}`), and `plan_review` should
  `green ... "select.pick[$idx].review"` with `{"file":..., "ok":true}`.
  Keep the retract/reject/provide lines as-is (unaffected by the rewire).

## 7. Edge cases / things to double-check while editing

- **Retraction still works under suffixed reduce.** In `research.yaml`,
  retracting a `gather.source[N]` must still drop that source out of
  `synth`'s reduce entirely (not require a `.verdict` for the retracted
  member) — this is existing engine behavior from PR #58 (§11 suffix-reduce
  rule operates over *surviving* members only), not something the example
  needs to special-case; just don't write example prose that implies
  otherwise.
- **`check`'s body wording** — since `synth` now genuinely waits on
  `.verdict`, make sure `check`'s body doesn't still describe retract as
  merely "dropping out of a parallel race" (the old framing) but as
  removing both the member and its `.verdict` requirement from the fan-in.
- **One suffix level only** — do not nest (`src[*].a.b` is a parse error
  per §11.x); both `.vetted` and `.review` are single-level suffixes, so
  this is not a risk here, just confirm no stray double-dot typos when
  writing the YAML.
- **Schema additions on `.vetted` / `.review`** are new relative to the
  draft (§4a, §4c) — if `lint`/`check` reject them for any reason (e.g. a
  produce-schema constraint this plan didn't anticipate), fall back to
  unscheduled/no-schema produces like the draft had, and note the deviation
  in the PR description; this is a contained, reversible fallback, not an
  escalation.
